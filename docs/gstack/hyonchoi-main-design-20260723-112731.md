# Design: Add 'UI Review' decision field for phase_6_2 skip signal (TODO-31)

## Problem Statement
`phases.yaml`'s `phase_6_2_qa` prompt already reads `UI Review: required` from a
TODO entry's Decisions line to decide whether to run QA or exit 0 as a no-op. But
`UI Review` is not documented anywhere: not in `sections/schema.md`'s Decisions
field definition, not in `sections/auto-research.md`'s derivation rules, not in
`SKILL.md`'s workflow/preamble, and not in `TODOS.md`'s own preamble blockquote.
New TODO entries created via `todos-manager --add` have no way to know this field
exists, so `phase_6_2_qa` has no skip signal to check on any new entry.

## Constraints
- Must mirror the existing `Security Review required/not-required` field exactly
  (same vocabulary, same auto-derivation-from-keywords pattern) — this is an
  established, working pattern with 4 touch points already.
- Backfilling ~30 existing TODO entries with a UI Review value is out of scope.
- No changes to `phases.yaml` — it already reads the field name correctly; a
  missing field is just treated as "no skip signal, run QA anyway," which is the
  correct fallback behavior.

## Premises
1. `UI Review` should be a new Decisions sub-field, structurally mirroring
   `Security Review` (same required/not-required vocabulary, same
   auto-derivation-from-keywords pattern). **Agreed.**
2. The four touch points are correct and complete: `sections/schema.md`,
   `sections/auto-research.md`, `SKILL.md`, and `TODOS.md`'s preamble
   blockquote. **Agreed.**
3. Backfilling ~30 existing TODO entries is explicitly out of scope — new
   entries only. **Agreed.**
4. `phases.yaml` already treats a missing `UI Review` line as "no skip signal,
   run QA anyway" — this change is purely additive documentation/tooling, no
   phases.yaml edit needed. **Agreed.**

## Approaches Considered

### Approach A: Mirror Security Review exactly (minimal viable) — CHOSEN
Add `UI Review` as a sibling field to `Security Review` in all four locations,
using the exact keyword list from TODO-31's own "What": ui, frontend, design,
visual, layout, component, css, style, dashboard, artifact, page, screen, modal,
form, navigation, button, icon, animation.
- Effort: S, Risk: Low
- Pros: fastest to ship; follows an established, working pattern exactly; no
  new abstractions
- Cons: two independent boolean fields will keep growing ad hoc if a third
  phase-skip signal appears later

### Approach B: Generalize to a "Phase Skip Signals" sub-block (ideal architecture)
Restructure Decisions to carry an extensible set of `<Phase>: required/not-required`
signals under one documented sub-schema with a single derivation-rule table keyed
by phase.
- Effort: M, Risk: Med
- Pros: scales cleanly if more optional-phase signals appear (TODO-24/29 lineage)
- Cons: larger diff than needed now; restructures the existing Security Review
  field's shape too, risking churn on ~30 existing entries' Decisions lines

### Approach C: Runtime keyword check, no schema change (lateral)
`phase_6_2_qa`'s own prompt runs the keyword check against title/summary at
runtime instead of reading a stored field.
- Effort: S, Risk: Med
- Pros: zero schema/doc changes, no backfill question ever comes up
- Cons: diverges from the established Security Review pattern; loses
  auditability of a human-visible/overridable decision in TODOS.md

## Recommended Approach
**Approach A.** It's what TODO-31 already specifies, mirrors a proven pattern,
ships today. Approach B's generalization is speculative — no second future
phase-skip signal is confirmed yet, so don't build for it (YAGNI). Approach C
sacrifices auditability for no real savings.

## Implementation Notes
Three file edits plus one verification test, all additive (no restructuring of
existing Security Review lines):

1. **`skills/todos-manager/sections/schema.md`** — extend the `Decisions:` row's
   example format to include `UI Review \`required/not-required\`` alongside
   `Security Review`, following the same backtick-delimited style.
2. **`skills/todos-manager/sections/auto-research.md`** — add a derivation rule
   row: `UI Review` → `required` if title/summary contains any of the keyword
   list above; else `not-required`. Add to the auto-research output template and
   the "signal defaulted" confidence-annotation note.
3. **`skills/todos-manager/SKILL.md`** — add `UI Review` to the editable
   Decisions fields list (workflow section) and to the remediation text
   ("Set key decisions: Priority, Effort, Phase, Branch, Test Coverage, Security
   Review, UI Review").
4. **`TODOS.md`** preamble blockquote — checked against the actual file
   (`TODOS.md` lines 1-10): the preamble only lists top-level entry fields
   (`What:`, `Why:`, `Decisions:`, etc.) and never enumerates Decisions
   sub-fields like `Security Review` or `Priority` today. So **no direct edit
   to `TODOS.md`'s preamble is needed** — `schema.md` is the sole source of
   truth for sub-field definitions, confirmed, not hedged.

5. **Verification test:** run `todos-manager --add` and enter a title
   containing a UI keyword (e.g. "Add dashboard filter component"). Confirm
   the auto-research synthesis output shows `UI Review: required` in the
   proposed Decisions line before the entry is written. This confirms the
   auto-research.md derivation rule from step 2 is wired correctly end-to-end.
