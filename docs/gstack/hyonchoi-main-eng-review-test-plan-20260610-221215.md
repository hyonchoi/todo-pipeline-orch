## Key Interactions to Verify
- `pipeline-watch --auto` selects the next eligible TODO per project per the
  priority/effort/unblock/order sort, isolating per-project parse failures.
- `pipeline-watch --merge <project> --todo <id>` performs Phase 9: e2e confirmation gate,
  semver bump, VERSION/CHANGELOG.md update, git merge, kanban clear, lock release.

## Edge Cases
- Circular `Depends on` between two TODOs in the same project — marked ineligible,
  Slack-notified once per cycle composition (not every tick).
- `[~]` (on-hold) TODO as a dependency — treated as unsatisfied, blocks dependents.
- Malformed TODOS.md in one project during `--auto` — isolated, other projects proceed,
  Slack notification names the broken project.
- Kanban sync failure (hermes unreachable) — outbox-queued (collapsed to
  latest-per-project), retried next tick, never blocks the pipeline.
- `git merge` conflict during Phase 9 after VERSION/CHANGELOG.md already written —
  `merge_status: failed` + `error`, lock stays held, `--merge` re-runnable after manual fix.
- Branch name collision on `-attemptN` — runner increments to next available N.

## Critical Paths
- [→E2E] Full `--auto` tick across 2+ projects: one with an eligible TODO selected and
  run through the phase loop with `set_active_task`/`update_phase` kanban calls; one with
  a dependency cycle (notified once); one with a malformed TODOS.md (isolated).
- [→E2E] Phase 9 merge flow: `ready_for_review` record → e2e confirmation → semver bump →
  VERSION/CHANGELOG.md write → git merge → `merge_status: merged` → `clear_active_task` →
  lock released.
