## Problem Statement

The tutorial, README, CHANGELOG, and superpowers plan all assume `pipeline-tick` exists as a Hermes command — but the Python code has no handler for it. The cron fires `pipeline-tick` every 5 minutes, but the command is not registered, so the pipeline never drives itself. The selection engine (`run_selection`) exists, the kanban module exists, and the phases config exists — but there is no command to tie them together.

## What Makes This Cool

The shift from Python-orchestrated subprocess spawning to **Kanban-as-Scheduler** is the key insight. Instead of `pipeline-tick` being a long-running process that blocks on phase execution, it becomes a SHORT registrar: select a TODO, register phases as Kanban tasks with parent-child dependencies, fire `hermes kanban dispatch`, and walk away. Hermes Kanban's native `--parent` flag and `dispatch` command handle the rest — reclaiming stale claims, promoting ready tasks, spawning workers. The pipeline stops being a monolithic orchestrator and becomes a thin layer on top of a general-purpose task graph engine.

## Constraints

- **Lock semantics:** `pipeline-tick` must not pick another TODO until all phases for the current TODO are fully completed. The lock is held based on kanban task status — before acquiring, check if a prior tick's kanban tasks are still in-flight.
- **Kanban model conflict:** The existing `HermesKanbanAdapter` in `kanban.py` manages a single active task per project (`set_active_task` → `update_phase` → `clear_active_task`). The new design creates multiple kanban tasks per TODO with `--parent` dependencies. **Resolution:** The tick command uses raw `hermes kanban` CLI directly (not through `HermesKanbanAdapter`). The existing adapter serves the old single-task model which will be retired once the kanban-as-scheduler path is fully operational.
- **Phase ordering:** Phases in `phases.yaml` are ordered by YAML list position (not phase key numbering — Phase 5 is missing). The `--parent` chain is built from list order: task at index 0 has no parent, task at index N has parent = task at index N-1.
- **Circuit breaker:** N consecutive no-progress ticks must back off cron and emit a Slack alert. The feedback loop from kanban outcomes must feed this.
- **Existing code reuse:** `decision.run_selection()`, `decision/context.build_context()`, `circuit.CircuitBreaker` all exist and must be used, not replaced.
- **Hermes kanban daemon:** The dispatcher runs in `hermes kanban daemon --interval 60`, not a separate cron. `hermes gateway` is a messaging gateway (Telegram, Discord) and does NOT run kanban dispatch.
- **ULID for tick IDs:** ULID is a 128-bit lexicographically sortable unique ID (like UUID but time-ordered). Requires `python-ulid` as a new dependency.

## Premises

1. `pipeline-watch tick` is a CLI subcommand. Hermes cron calls `pipeline-watch tick` every 5 minutes.
2. `pipeline-tick` is SHORT — acquire lock, select, register kanban tasks, dispatch, release lock. Does NOT block on phase execution.
3. Phases become kanban tasks with `--parent` dependencies. `hermes kanban dispatch` (in gateway) promotes and spawns workers when parents complete.
4. Kanban task completion feeds back into the decision/outcome store and circuit breaker.
5. ULID for tick IDs (requires `python-ulid` dependency).
6. `phases.yaml` becomes a kanban task registration template — its prompts define kanban task bodies.

## Approaches Considered

### Approach A: Thin Registrar

`pipeline-watch tick` reads phases.yaml, creates kanban tasks with `--parent` deps, fires dispatch. Tick takes ~10 seconds. `phases.yaml` becomes a kanban task registration template. Minimal code — new tick subcommand + kanban task creation logic. **No `observe_outcomes` call** — kanban task completion is not fed back to the decision store, so the circuit breaker cannot observe progress.

- Effort: S
- Risk: Low
- Pros: Fastest to ship. Minimal risk.
- Cons: No state sync between kanban and decision store. Circuit breaker has no data.

### Approach B: Full Lifecycle with Feedback (Chosen)

Same as A but adds bidirectional state sync: kanban task completion feeds back into decision/outcome store, circuit breaker observes progress, and stale kanban tasks are reconciled on tick. More robust, more code.

- Effort: M
- Risk: Medium
- Pros: Complete feedback loop. Circuit breaker works. State is consistent across kanban and decision stores.
- Cons: More code, more edge cases.

### Approach C: Kanban Decompose

Use `hermes kanban decompose` to let the LLM break the picked TODO into phase subtasks dynamically. More flexible — adapts to TODO complexity. Less deterministic than phases.yaml template.

- Effort: M
- Risk: High
- Pros: Adapts to TODO complexity. No phases.yaml maintenance.
- Cons: Non-deterministic. Hard to test. Phase structure can drift between ticks.

## Recommended Approach

**Approach B: Full Lifecycle with Feedback.** The circuit breaker, decision store, and kanban state must stay in sync — otherwise the pipeline loses observability and the no-progress detection (which backs off cron and alerts) cannot function. The extra code in Approach B is the price of a system that can actually be operated, not just demoed.

## Success Criteria

1. `pipeline-watch tick` runs, selects a TODO, registers kanban tasks, and exits in under 30 seconds.
2. `hermes kanban daemon` (running continuously) automatically promotes and spawns workers as phase parents complete.
3. When all phases for a TODO complete, the next tick observes this and writes outcome sidecars to the decision store; the circuit breaker resets its counter.
4. The lock prevents `pipeline-watch tick` from picking another TODO while the current TODO has in-flight phases.
5. N consecutive no-progress ticks trigger cron backoff and Slack alert.

## Schema Extension

The existing `Outcome` type in `schema.py` is `Literal["in_flight", "merged", "failed_at_phase_N", "discarded", "killed_by_operator", "failed_to_spawn"]`. The new design adds `phase_complete` and `all_phases_complete` outcomes to the JSONL file (not the one-time outcome sidecar). No change needed to the `Outcome` literal — the JSONL file uses free-form outcome strings.

## Race Condition: Lock → Kanban Check

The tick lock is file-based (atomic mkdir), but the "all phases complete" check queries kanban state. Between the check and the lock release, another tick could start. Mitigation: the tick flow checks `all_phases_complete()` **before acquiring the lock**, not after. If a prior tick's kanban tasks are still in-flight, the tick skips selection without acquiring the lock — no critical section to race on. If a prior tick has no kanban tasks (e.g., picked=None), the tick acquires the lock and proceeds normally.

## Distribution Plan

The deliverable is a Python CLI subcommand — no separate binary. Distribution is via `uv` editable install of the `hermes-pipeline` package. The `pipeline-watch` entry point already exists (defined in pyproject.toml); the `tick` subcommand is added to its argparse subparsers.

### Step 3: Implement kanban task registration (`hermes_pipeline/kanban_tasks.py`)

- `register_todo_phases(todo_id, tick_id, board_slug, project_dir, phases)` — reads phase config, creates kanban tasks with `--parent` deps. Uses raw `hermes kanban create` CLI directly (not through `HermesKanbanAdapter`). Task body format: first line is a JSON header, followed by the rendered phase prompt:
  ```json
  {"tick_id":"01J6...","phase_key":"phase_2_autoplan","todo_id":"TODO-10","project_slug":"todo-pipeline-orchestrator"}
  ```
  The rendered phase prompt comes from `_render_phase_prompt` in phases.py. Tasks use `--idempotency-key <tick_id>:<phase_key>` for dedup on retry. Workspace: `--workspace dir:<project_dir>`. Goal mode: `--goal --goal-max-turns <turns_from_phases_yaml>`. The `--parent` chain is built from YAML list order: task at index 0 has no parent, task at index N has parent = task at index N-1. On mid-registration failure: archive already-created tasks via `hermes kanban archive` and raise (the tick aborts; idempotency keys ensure the next cron fire retries cleanly).
- `get_todo_kanban_status(board_slug, tick_id)` — query kanban for all tasks of a TODO tick (parse tick_id from task body JSON header), return `{phase_key: status}` map
- `observe_outcomes(state_dir, tick_id, status_map)` — read kanban task completion, write outcome sidecars to decision store. **Outcome persistence model:** The existing `store.append_outcome()` writes `.hermes/outcomes/<tick_id>.json` once (raises `FileExistsError` on retry). For multi-phase outcomes, use a separate file: `.hermes/outcomes/<tick_id>-phases.json` — a JSONL file where each line is one phase outcome. This avoids the FileExistsError conflict with the existing one-time write model.

**Bidirectional state sync (Approach B detail):**

Direction 1 — Decision → Kanban: When `run_selection` picks a TODO, `register_todo_phases` creates kanban tasks. The tick_id, phase_key, and rationale from the decision are embedded in the kanban task body (JSON header) so workers have context and the outcome observer can map tasks back to phases.

Direction 2 — Kanban → Decision Store: After phase completion, `observe_outcomes` runs at the end of the tick. It queries kanban for the tick's task status map (extracts tick_id and phase_key from the JSON header in the task body):
- If a phase task is `done`: append `{"outcome": "phase_complete", "phase_key": "...", "completed_at": "..."}` to `.hermes/outcomes/<tick_id>-phases.json` (JSONL)
- If a phase task is `failed`: append `{"outcome": "failed_at_phase_<key>", "detail": {"error": "..."}}` — same shape as `phases.py` failure sidecars
- If all tasks are done: append a terminal outcome `{"outcome": "all_phases_complete", "completed_at": "..."}` — the decision is fully resolved

**Note:** The decision store is updated on the NEXT tick (not immediately when a phase completes). This means up to 5 minutes of stale state, which is acceptable for an orchestrator that runs on 5-minute intervals. For real-time outcome tracking, use `hermes kanban watch` or the kanban event stream.

Circuit breaker integration: The tick's `observe_outcomes` feeds `circuit.CircuitBreaker.observe(picked, counts_as_no_progress)`. A tick counts as no-progress if: (a) picked is None (selection stall), or (b) a phase failed. Tasks stuck in `running` for > `max_phase_timeout_min` are NOT counted as no-progress — the kanban daemon reclaims them. A tick that made progress (phase completed) resets the counter.

**Approach A vs B:** Approach A (thin registrar) would skip `observe_outcomes` entirely — no direction 2 sync, no circuit breaker feedback. The tick flow would omit the `observe_outcomes` call after step 6. Approach B adds it.

**Lock completion check:** `all_phases_complete(board_slug, tick_id)` queries kanban for the tick's tasks. Returns True iff every task is in a terminal status (`done`, `failed`, `archived`). Used by the tick flow: before acquiring the lock for a new selection, check if a prior tick's tasks are complete. If a prior tick exists and tasks are still in-flight, skip selection and release (lock not acquired — another tick's work is ongoing).

**Board slug derivation:** `board_slug = project_slug`. All kanban commands use `--board <project_slug>`. One board per project. Board is auto-created by `hermes kanban create` if it doesn't exist (the Hermes CLI handles board initialization).

### Step 4: Add `tick` subcommand to CLI (`hermes_pipeline/cli.py`)

```
pipeline-watch tick <project>
```

Command naming: The Hermes cron entry fires `hermes cron set pipeline-tick '*/5 * * * *'`. When the cron triggers, the Hermes command `pipeline-tick` (defined in the Hermes config repo) invokes `pipeline-watch tick <project>` via the Python CLI. The cron label is `pipeline-tick`; the Python subcommand is `tick`. They resolve to the same execution path.

Flow:
1. `prior_tick_id = read_prior_tick_id(state_dir)` — read `.hermes/current_tick_id.txt` if it exists. If a prior tick exists, `all_phases_complete(board_slug, prior_tick_id)` — check if its kanban tasks are still in-flight. If yes, skip: log "prior tick in-flight, skipping" and exit.
2. `acquire()` — get tick lock, mint ULID tick_id, persist to `.hermes/current_tick_id.txt`
3. `build_context()` — assemble SelectionContext from TODOS.md, in-flight set, recent decisions, kanban snapshot
4. `run_selection()` — agent call, SHA pin, persist decision
5. If `picked` is None: observe circuit breaker (no_progress=True), release lock, exit
6. If `picked` is TODO-N: `register_todo_phases()` — create kanban tasks with parent deps. On mid-registration failure: archive already-created tasks, raise. The tick aborts and will be retried on the next cron fire (idempotency keys prevent duplicate tasks).
7. Observe circuit breaker
8. Release lock (TickLock context manager handles cleanup)
9. Exit

Note: `hermes kanban dispatch` is NOT called from the tick. The kanban daemon runs dispatch every 60 seconds. Calling dispatch from the tick would be redundant and could race with the daemon.

### OV-1: `build_in_flight` must query kanban status, not file markers

**Problem:** The current `build_in_flight()` in `decision/context.py` reads file markers (`ready_for_review/` and `phase_started/` directories) to determine which TODOs are in-flight. With kanban-as-scheduler, the source of truth for "what's running" is the kanban board, not file markers. Relying on file markers creates a split-brain where the selection engine sees stale state.

**Resolution:** Add a `build_in_flight_from_kanban(board_slug)` function that queries `hermes kanban list --board <slug> --json` and extracts the set of TODO IDs with non-terminal kanban tasks (tasks in `created`, `ready`, `running` status). The function parses the JSON header in each task's body to find `todo_id`.

The existing `_kanban_snapshot()` in `context.py` already does this — it's just not wired into `build_in_flight`. The updated flow:

```python
def build_in_flight(state_dir: Path, board_slug: str, *, max_phase_timeout_min: int) -> list[str]:
    """Compute in-flight set from kanban state, falling back to file markers."""
    kanban_in_flight = _kanban_in_flight_ids(board_slug)
    if kanban_in_flight is not None:
        return sorted(kanban_in_flight)
    # Fallback: file markers (for pre-kanban state or if kanban CLI fails)
    return build_in_flight_from_files(state_dir, max_phase_timeout_min=max_phase_timeout_min)

def _kanban_in_flight_ids(board_slug: str) -> set[str] | None:
    """Extract TODO IDs with in-flight kanban tasks. Returns None on CLI failure."""
    snapshot = _kanban_snapshot(board_slug)
    if snapshot is None:
        return None
    result = set()
    for task in snapshot.get("tasks", []):
        if task.get("status") in ("created", "ready", "running"):
            # Extract todo_id from JSON header in task body
            body = task.get("body", "")
            first_line = body.split("\n")[0]
            try:
                header = json.loads(first_line)
                todo_id = header.get("todo_id")
                if todo_id:
                    result.add(todo_id)
            except (json.JSONDecodeError, IndexError):
                pass  # Task without JSON header — skip
    return result
```

**Backward compatibility:** Falls back to file markers if kanban CLI is unavailable (migration period).

### OV-2: Circuit breaker needs JSONL outcome integration

**Problem:** The current `CircuitBreaker.observe(picked, counts_as_no_progress)` requires the caller to determine whether a tick made progress. With the JSONL outcome file (`.hermes/outcomes/<tick_id>-phases.json`), the circuit breaker should be able to read outcomes directly and derive its own judgment.

**Resolution:** Add an `observe_from_outcomes()` method that reads the JSONL file and determines no-progress status:

```python
def observe_from_outcomes(self, state_dir: Path, prior_tick_id: str, current_picked: str | None) -> None:
    """Observe circuit breaker state from JSONL outcome file."""
    phases_file = state_dir / ".hermes" / "outcomes" / f"{prior_tick_id}-phases.json"
    if not phases_file.exists():
        # No outcome file — prior tick didn't register phases (Approach A or picked=None)
        # Fall back to explicit observe
        return self.observe(picked=current_picked, counts_as_no_progress=True)

    lines = phases_file.read_text().strip().split("\n")
    outcomes = [json.loads(line) for line in lines if line.strip()]

    has_phase_complete = any(o.get("outcome") == "phase_complete" for o in outcomes)
    has_all_complete = any(o.get("outcome") == "all_phases_complete" for o in outcomes)
    has_failure = any(o.get("outcome", "").startswith("failed_at_phase_") for o in outcomes)

    if has_all_complete:
        # Pipeline fully completed a TODO — definite progress, reset counter
        return self.observe(picked=current_picked, counts_as_no_progress=False)
    if has_phase_complete:
        # At least one phase completed — progress, reset counter
        return self.observe(picked=current_picked, counts_as_no_progress=False)
    if has_failure:
        # Phase failed — no progress, increment counter
        return self.observe(picked=current_picked, counts_as_no_progress=True)

    # No outcomes yet — tick is still in-flight, don't count as no-progress
    return self.observe(picked=current_picked, counts_as_no_progress=False)
```

The tick flow calls `observe_from_outcomes()` at step 7, passing the prior tick's ID and current selection result. This eliminates the need for the caller to manually determine `counts_as_no_progress`.

### OV-3: Phase timeout enforcement mechanism

**Problem:** If a phase worker gets stuck (e.g., infinite loop, hung on user input), the kanban daemon's stale-claim reclaim handles it — but the interval and timeout need explicit configuration. The current `max_phase_timeout_min` in the circuit breaker config is used for file marker staleness, not kanban task timeouts.

**Resolution:** The phase timeout is enforced by the kanban daemon's `--stale-claim-timeout` flag. Add a `phase_timeout_min` configuration to `phases.yaml`:

```yaml
pipeline:
  phase_timeout_min: 30  # Default: 30 minutes per phase
  goal_max_turns: 20     # Default max turns per phase goal
```

The `register_todo_phases()` function passes `--stale-claim-timeout` to `hermes kanban create`:

```python
cmd = [
    "hermes", "kanban", "create",
    "--board", board_slug,
    "--workspace", f"dir:{project_dir}",
    "--goal", f"--goal-max-turns", str(phase.get("goal_max_turns", 20)),
    "--idempotency-key", f"{tick_id}:{phase_key}",
    "--stale-claim-timeout", f"{phase_timeout_min}m",
    # ...
]
```

If `--stale-claim-timeout` is not supported by the hermes kanban CLI, the timeout is enforced by the `hermes kanban daemon --interval 60` which checks for stale claims (task in `running` for > `--interval * stale_threshold`). The existing `_phase_started_ids()` stale-sweep logic is the fallback.

**On timeout:** The kanban daemon reclaims the task (sets status to `ready`), and the next dispatch cycle picks it up again. If a phase fails N times (tracked via retry count in the kanban task metadata), it transitions to `failed` status — the tick's `observe_outcomes()` writes `failed_at_phase_<key>` to the JSONL file, and the circuit breaker counts it as no-progress.

**Retry limit:** Default 2 retries per phase. After 2 retries, the phase is marked `failed` and the circuit breaker is notified. Prevents infinite reclaim-retry loops.

### OV-4: Operator resume mechanism for failed phases

**Problem:** When a phase fails (after retries or due to a hard error like a merge conflict), the pipeline needs a mechanism for the operator to intervene and resume from the point of failure — not restart from phase 1.

**Resolution:** The kanban model naturally supports this. When a phase task fails, the operator can:

1. **Fix and retry:** `hermes kanban retry <task_id>` — resets the task to `ready` status, dispatch picks it up on the next cycle.

2. **Skip and promote:** `hermes kanban promote --skip <task_id>` — marks the task as `done` (skipped) so the next phase in the `--parent` chain becomes unblocked.

3. **Re-register from phase N:** A new `pipeline-watch resume <project> --from-phase <phase_key>` subcommand that re-registers only the failed and subsequent phases:
   ```
   pipeline-watch resume todo-pipeline-orchestrator --from-phase phase_4_review
   ```
   This reads the prior tick's kanban task status, archives any tasks still in a failed state, and re-creates them with new `--idempotency-key` values (appending `:retry-1`).

**Implementation note:** `pipeline-watch resume` is a follow-up command, not required for TODO-10. The `hermes kanban retry` and `hermes kanban promote --skip` CLIs are sufficient for V1 operator intervention.

**Circuit breaker impact:** When the operator resumes a failed phase, the circuit breaker counter should reset — the operator's intervention counts as progress. The `pipeline-watch resume` command writes a `{"outcome": "operator_resumed", "phase_key": "..."}` entry to the JSONL file, which the circuit breaker interprets as `counts_as_no_progress=False`.

### OV-5: `pipeline-tick` Hermes command registration

**Problem:** The design doc describes the tick flow and the cron entry (`hermes cron set pipeline-tick '*/5 * * * *'`), but doesn't specify how the Hermes command `pipeline-tick` is registered. Hermes cron invokes commands by name — the command must be defined in the Hermes config or skill system for the cron to resolve it.

**Resolution:** Register `pipeline-tick` as a Hermes command via the Hermes config in the project's `.hermes/` directory:

```yaml
# .hermes/commands/pipeline-tick.yaml
name: pipeline-tick
description: "Tick the pipeline — select a TODO and register kanban phases"
executor: shell
command: "uv run pipeline-watch tick {{project}}"
triggers:
  - cron: "*/5 * * * *"
args:
  - name: project
    required: true
    default: "{{.ProjectSlug}}"
```

The `{{project}}` variable is resolved from the project slug. Alternatively, if Hermes commands are defined in the Hermes config repo (not project-local), the command definition lives there:

```yaml
# hermes-config/commands/pipeline-tick.yaml
name: pipeline-tick
executor: shell
command: "uv run pipeline-watch tick {{.ProjectSlug}}"
cron: "*/5 * * * *"
```

**Verification:** Run `hermes commands list` after registration to confirm `pipeline-tick` is available.
