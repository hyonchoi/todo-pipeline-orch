## Problem Statement

`phase_2b_plan_gate` has already been removed from `phases.yaml` (TODO-24 confirmed —
verified via grep, zero matches for `phase_2b`/`plan_gate` in the current profile). That
leaves an entire subsystem of dead code behind it: `approve_plan.py`, the `approve-plan`
CLI subcommand, and gate-status logic in `gates.py`/`gate_state.py`/`kanban_tasks.py`
(TODO-26). Separately, `harness.py`'s `--kanban null` mode drives phases through a local
scheduler (`PipelineRunner` in `runner.py` → `phases.run` → `_invoke_hermes`/
`_invoke_review_phase`) that never runs in production — production dispatches phases
exclusively through kanban tasks (`register_todo_phases` renders the full prompt at
creation time; nothing re-dispatches dynamically). This violates the harness-must-mirror-
production invariant and left `review_phase.py`'s PRE/POST split, `phases.py`'s marker
machinery, and `cmd_kill` all effectively already-dead in production (TODO-27).

The user asked to handle both TODOs in one office-hours session, since their deletion
targets overlap in `phases.py` and `cli.py`.

## Approaches Considered

### Approach A: Single-PR full deletion (bottom-up dependency order)
Summary: One branch/PR deletes everything in dependency order — cli.py subcommands first,
then harness.py null-mode + runner.py + watcher.py, then review_phase.py PRE/POST, then
phases.py's gate dispatch + markers, then gates.py/gate_state.py/kanban_tasks.py plan-gate
branch, then state.py helpers, then delete/prune the affected test files.
Effort: L
Risk: Med (large diff, but every symbol grep-verified with zero live callers)
Pros: One coherent branch matches "these files all touch the same dead subsystem";
avoids TODO-27 half-deleting phases.py while TODO-26 edits the same file mid-flight.
Cons: Big diff for a single PR review.

### Approach B: Two sequential PRs, same session (26 first, then 27)
Summary: PR1 = original TODO-26 scope only. PR2 = TODO-27 scope, branched after PR1 merges.
Effort: M + L (sequenced)
Risk: Low per-PR
Pros: Smaller reviewable diffs; TODO-26 ships independently if TODO-27 stalls.
Cons: phases.py's gate-dispatch block (lines 320-347) belongs to both TODOs and has to
be arbitrarily assigned to one PR anyway; more process overhead than the "handle both
together" request implied.

## Recommended Approach

**Approach A** — single combined PR. The user explicitly asked to handle both TODOs in
this session; the dependency graph between the two deletions overlaps in `phases.py` and
`cli.py` in a way that can't be cleanly split without one PR temporarily leaving stray
dead references; and every deletion candidate has been grep-verified this session to have
zero live callers, so risk is low despite the diff size.

## Amended Deletion Scope (from /plan-eng-review, 2026-07-22)

The interactive eng review + outside voice found the plan's own "zero non-test callers"
verification was incomplete in 6 places. All amendments below were confirmed with the
user via AskUserQuestion — this section supersedes the original Verification/Success
Criteria wherever they conflict.

### Additional deletion targets

1. **`gates.maybe_plan_gate_ready` (gates.py:322) + its `cli.py:1133-1141` call site +
   `PLAN_GATE_PHASE_KEY`.** The original plan's claim that `PLAN_GATE_PHASE_KEY` is "used
   only by gate_state.py and gates.py's own prior-tick handler" was wrong — it's called
   from `cli.py`'s tick loop on every tick with a `prior_tick_id`. It's a no-op in
   practice (early-returns at gates.py:350 once `phase_2b_plan_gate` never appears in
   `status_map`) but not dead code by the plan's own "zero callers" definition, and
   deleting `PLAN_GATE_PHASE_KEY` without touching the call site raises a `NameError` on
   the next tick. **Delete the whole chain together.**

2. **`hermes_pipeline/merge.py`** (214 lines: `run_phase9`, `make_default_bump_fn`,
   `default_bump_fn`, `default_confirm_fn`) — orphaned once `cli.py`'s `_cmd_merge` and
   `merge` subcommand are deleted (its only caller). Delete the whole file plus
   `tests/test_merge.py` (372 lines) and `tests/test_chaos.py` (244 lines, run_phase9
   retry/chaos tests) entirely. Delete `tests/test_cli_kill.py` (220 lines) wholesale
   rather than "prune" — it tests only `cmd_kill`/`_cmd_kill`, both being deleted.
   *(This is TODO-29's core scope — see "TODOS.md updates" below.)*

3. **`hermes_pipeline/status.py`** (187 lines: `StatusRow`, `collect_pending`,
   `format_table`, `_age`) backs the `status` CLI subcommand (`cli.py:726` `_cmd_status`).
   Deleting `ReadyForReview`'s write path (phases.py/runner.py, both already slated for
   deletion) means nothing will ever populate `ready_for_review/*.json` again — `status`
   would silently become a permanently-empty table, a live user-facing regression, not
   dead code. Delete `status.py` + the `status` subcommand + `tests/test_status.py`
   entirely rather than leave a command that always prints nothing.

4. **`hermes_pipeline/review_phase.py`** (246 lines) — the plan named 4 functions to
   delete (`capture_pre_review_state`, `finalize_review`, `run_pytest`, `commit_all`) but
   every other symbol in the file (`restore_worktree`, `write_review_artifacts`,
   `PytestResult`, `PreReviewState`, `_verify_review_success`, etc.) is reachable only
   from those 4 functions' own call graph. The module's only external caller
   (`phases.py`'s `_invoke_review_phase`) is itself being deleted. Delete the whole file
   plus `tests/test_review_phase.py` (282 lines) wholesale.

5. **`NullKanbanAdapter`** (kanban.py:86, ~30 lines) — the plan's Constraints section
   explicitly said this stays, citing `cli.py:686`'s `_cmd_merge` fallback as a live
   caller. But `_cmd_merge` is deleted per item 2 above, and `harness.py:713`'s other
   usage is inside the `--kanban null` branch already in original scope for deletion.
   Once both go, `NullKanbanAdapter` has zero remaining callers in `hermes_pipeline/*.py`
   — the plan's own stated reason for keeping it no longer holds. Delete the class and
   prune its dedicated `TestNullKanbanAdapter` class in `tests/test_kanban.py` (the rest
   of that 812-line file — protocol/outbox/`HermesKanbanAdapter` tests — stays).

6. **`hermes_pipeline/decision/context.py:159-163`'s `in_flight_ids()` fallback** — when
   kanban lookup fails, this reads `ready_for_review/*.json` (`_rfr_ids`, line 25) and
   `phase_started/*` markers (`_phase_started_ids`, line 51) directly off disk to avoid
   re-picking a TODO that's already in flight. Both directories are populated only by
   code this PR deletes (`phases.py`'s `write_ready_for_review` + `_write_marker`). Post-
   deletion this fallback silently becomes a permanent no-op: during any kanban outage,
   the system would think nothing is in flight and could double-pick a TODO. This is a
   genuine **behavior change**, not pure deletion — collapse `in_flight_ids()` to rely
   solely on the kanban lookup, remove `_rfr_ids`/`_phase_started_ids` and the
   `| set(...)` union, and update `tests/test_decision_context.py` +
   `tests/test_decision_context_edge.py` accordingly.

Scope grew roughly 3-4x (3 files → 7 files deleted, plus doc updates, plus the
`decision/context.py` behavior fix) but stays as **one combined PR**. The new findings
are discovered via the same dependency graph the original single-PR argument was built
on — splitting now would mean re-deriving the same cross-file dependencies across PR
boundaries. The diff is large but mechanical (deletions + doc updates), which reviews
faster than an equivalent-sized feature PR.
