## Problem Statement
`hermes_pipeline/harness.py` (804 lines) drives a mock end-to-end pipeline-watch
cycle for testing, but there's no structured way to verify it actually exercises
the real pipeline-watch production code paths versus a parallel reimplementation.
TODO-21 (blocked on this TODO) will refactor the harness to reuse production
modules directly — but "reuse production code" has no acceptance criteria today.
This session produces that acceptance criteria: a checklist mapping each
behavioral transition in one automated pipeline-watch cycle to the production
functions that must fire and the state change that must result.

## What Makes This Cool
It turns "the harness should use production code" from a vibe into something
checkable. Once this checklist exists, TODO-21 isn't "refactor harness.py and
hope it's still correct" — it's "make these N transitions pass their spy
assertions, one at a time." The checklist also outlives TODO-21: it becomes the
regression gate that keeps harness.py honest as pipeline-watch evolves, so a
future change to `circuit.py` or `kanban_tasks.py` can't silently drift out of
sync with what the harness pretends to test.

## Constraints
- Ground truth is pipeline-watch's own code (`hermes_pipeline/*.py`), not the
  current harness.py and not just the docs — the docs (`hermes-state-machine.md`,
  `reference-kanban-as-scheduler.md`) are useful orientation but may lag the code.
- Scope is one automated pipeline-watch cycle in the **kanban-as-scheduler path**
  (`kanban_mode == "hermes"`, i.e. `harness.py::_poll_kanban_phases` and its
  callees) — this is the path TODO-20/21/22 all target. harness.py's other path
  (`PipelineRunner` + null kanban adapter) is a separate, out-of-scope concern.
  Scope covers: tick lock → phase run → gate check → kanban sync →
  convergence/circuit observation → completion (ready_for_review / merged /
  failed / killed). Multi-project scan and Slack alert delivery are outside a
  single cycle and out of scope for this checklist.
- harness.py itself is not the target of verification — it's a thin wrapper that
  TODO-21 will make simulate one pipeline-watch cycle without human input. This
  checklist verifies pipeline-watch behavior; harness compliance with it is a
  downstream, later-verified property. (Earlier draft carried a marker-lifecycle
  exception for `.hermes/phase_started/`, but `_write_marker`/`_delete_marker`
  only fire via `phases.run()`, which only fires via `_dispatch_phase` in the
  non-kanban `PipelineRunner` path — out of scope per the kanban-as-scheduler
  restriction below. Dropped as inapplicable to this scope.)
- Checklist items are call-spy assertions (wrap/monkeypatch the real production
  function, assert it was called) paired with a resulting-state assertion — not a
  static import/grep audit and not a full golden-cycle fixture.
- Checklist rows are prose-only (Trigger | Pre-state | Post-state | Production
  function(s) | State assertion in plain English) — no literal code snippets in
  the doc, matching the style of the existing `docs/hermes-state-machine.md`
  table.
- The new checklist doc stays separate from `docs/hermes-state-machine.md`, with
  rows cross-referencing the existing table's row numbers where transitions
  overlap, rather than duplicating content.
- **Where harness.py's current behavior has no production counterpart today**
  (e.g. convergence detection currently uses harness-local `ConvergenceDetector`/
  `_ConvergenceMonitor` instead of `circuit.py`'s `CircuitBreaker`/
  `observe_from_outcomes`; `_auto_complete_gate_tasks()` has no direct production
  equivalent), the checklist still names the CORRECT production target that
  TODO-21 should route through — not what harness.py calls today. Surfacing that
  gap is the checklist's job, not a reason to omit the row.

## Premises
1. Ground truth for "correct pipeline-watch behavior" is the current code in
   `hermes_pipeline/*.py`, not `docs/hermes-state-machine.md` or
   `docs/reference-kanban-as-scheduler.md` (which may be stale/aspirational) —
   AGREED.
2. Checklist scope is one automated cycle only (phase execution → gate → kanban →
   convergence → completion). Multi-project scan, Slack alerts, and preflight
   startup checks are out of scope — AGREED.
3. Checklist items are organized by state-machine transition/behavior (e.g. "gate
   blocked → tick lock held until override", "convergence detected → halt"), not
   flat one-per-function — richer than call-presence, closer to true behavior
   verification. Each item names the production function(s) involved AND the
   resulting state assertion — AGREED.
4. The checklist lives as a markdown doc
   (`docs/checklist-harness-production-coverage.md`) with a `Test link` column per
   row, rather than being encoded directly as a test file's structure — AGREED.

## Approaches Considered

### Approach A: One-cycle transition table with spy assertions written now
Summary: Populate the full doc AND write the spy test for every row this
session, against the current (pre-TODO-21) harness.py.
Effort: M
Risk: Low
Pros:
- Immediately "executable" — every checklist row has a real, runnable assertion
- Failing tests concretely prove today's gap (harness reimplements logic) rather
  than asserting it in prose
Cons:
- Some spy tests will need rework once TODO-21 restructures harness.py's
  internals, since they're written against call sites that are about to move
- Exceeds TODO-22's own stated effort estimate (`S`) — blurs into TODO-21's work

### Approach B: Doc first, tests deferred to land with TODO-21 (chosen)
Summary: Finalize the transition table now with every row fully specified
(trigger, production functions, state assertion) but leave the `Test link`
column as `TODO(TODO-21)` placeholders. Each spy test gets written and linked in
the same TODO-21 PR that makes the harness actually call that transition's
production functions.
Effort: S
Risk: Low
Pros:
- Matches TODO-22's own effort estimate (`S`) — a spec, not a test suite
- Avoids throwaway spy scaffolding written against harness.py internals TODO-21
  is about to restructure
- Checklist and refactor land together, transition by transition, so nothing
  gets marked "done" without its test existing
Cons:
- Checklist isn't fully "executable" the moment TODO-22 closes — it becomes real
  incrementally during TODO-21
- Requires discipline during TODO-21 to actually pair each transition with its
  test rather than batching the refactor and leaving link columns stale

### Approach C: Shared golden-cycle fixture + full harness (not chosen)
Summary: Build one shared pytest fixture that runs a complete synthetic
pipeline-watch cycle using real production modules end-to-end, with every
checklist item asserting against calls captured during that one fixture run.
Effort: L
Risk: Med
Pros:
- Fast regression suite long-term — one fixture run backs many assertions
- Forces early validation of the "harness = thin wrapper" shape TODO-21 targets
Cons:
- Effectively starts TODO-21's core refactor inside this session — scope creep
- Edge-case transitions (timeout, kill, prompt SHA mismatch) don't fit one
  deterministic fixture path anyway, undermining the "one fixture" premise

## Recommended Approach
Approach B. TODO-22's own `Decisions` field specifies Effort `S` — a fully
code-grounded spec with deferred test links satisfies "acceptance criteria for
TODO-21" without pulling TODO-21's refactor work forward. It also naturally
enforces "no transition ships without its test" as a per-PR discipline during
TODO-21, rather than a big-bang test suite that risks partial rework.

## What already exists
- `docs/hermes-state-machine.md` — 16-row tick-lock-level transition table.
  Reused via cross-reference, not duplicated (see Constraints).
- `hermes_pipeline/circuit.py::CircuitBreaker`/`observe_from_outcomes` — exists,
  but harness.py's current convergence path does NOT call it; harness uses
  local `ConvergenceDetector`/`_ConvergenceMonitor` instead. The checklist
  targets `circuit.py`'s functions as the correct destination anyway (per
  Constraints) — this gap is exactly what TODO-21 must close.
- `hermes_pipeline/gates.py::check_gate_status` — exists; harness's
  `_auto_complete_gate_tasks()` has no direct production counterpart today.
  The checklist's row for this transition must name the gate-completion
  function TODO-21 should route through, resolved during doc-writing (open
  item carried into Next Steps).

## Success Criteria
- `docs/checklist-harness-production-coverage.md` exists with one row per
  in-scope transition (tick lock → phase run → gate → kanban → convergence →
  completion), each row naming: trigger, pre/post state, production function(s)
  that must be called, and the state assertion that proves it happened.
- Every row's production functions are verified against current
  `hermes_pipeline/*.py` code (not assumed from docs).
- TODO-21 can be scoped/estimated directly from this checklist's row count and
  transition boundaries.
