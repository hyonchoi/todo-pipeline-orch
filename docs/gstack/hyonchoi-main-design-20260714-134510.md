# Design: TODO-18 — `--revise` Subcommand for todos-manager

Mode: Builder

## Problem Statement

`--audit` and `--convert` are report-only. When `--convert` migrates a header-based TODOS.md to canonical format, the required fields (What, Why, Decisions) may be empty or weak because `--convert` only transforms structure, not content. After an audit surfaces missing fields, users must manually edit TODOS.md — there's no skill-driven workflow to fix the gaps.

## What Makes This Cool

Closes the audit-to-fix loop: run `--audit` to find broken entries, then `--revise` to fix them with AI pre-filled suggestions. The auto-research phase from `--add` already knows how to derive What/Why/Decisions from git log, design docs, and related TODOs. Reusing it for existing entries means one command fills in the blanks.

## Constraints

- Only revises **active** entries in TODOS.md. TODOS-archive.md is append-only — completed TODOs are never modified.
- Effort S: one new section file (`sections/revise.md`) + SKILL.md wiring.
- Must reuse auto-research from `sections/auto-research.md`, not reimplement.

## Premises

1. `--convert` only transforms format, not content — entries with missing required fields are a real post-convert state. **Confirmed by user.**
2. The auto-research phase from `--add` can be reused to pre-fill missing fields in existing TODOs (same codebase signals: git log, design docs, related TODOs).
3. Archived entries in TODOS-archive.md should never be modified. **Confirmed by user.**
4. User selects a TODO by ID — the skill doesn't auto-select.
5. The confirm/edit gate from `--add` (batch reply: "confirm" or "field: new value") is the right UX for `--revise` too.

## Approaches Considered

### Approach A: Targeted Revise (Chosen)

User picks a TODO-N -> scan for missing/weak fields -> run auto-research scoped to those gaps -> batch confirm/edit gate -> write back to TODOS.md. One entry at a time.

**Effort:** S (~1 new section file + SKILL.md wiring)
**Risk:** Low (reuses 90% of --add pipeline)

### Approach B: Revise with Bulk Mode

Adds --bulk flag: runs audit internally, walks through each failing entry one-by-one.

**Effort:** M
**Risk:** Med (new state machine, no existing pattern to reuse)

### Approach C: Revise with Field Selector

User picks TODO-N AND specifies which fields to revise.

**Effort:** S (A + flag parsing)
**Risk:** Low, but marginal benefit over A's batch edit gate

## Recommended Approach

**Approach A: Targeted Revise** — smallest diff that closes the audit-to-fix loop. Bulk and field-selector can be added later if evidence of need emerges.

## Workflow Specification

New section: `sections/revise.md`. Referenced from SKILL.md `--revise` subcommand block and the section index.

### Steps

1. **Validate context:** TODOS.md exists? If not, prompt `--init`.
2. **Prompt for TODO-N:** "Enter the TODO ID to revise (e.g. TODO-5):"
   - Validate: ID must match `TODO-<digits>` pattern.
   - Validate: entry must exist in TODOS.md (active entries only).
   - **Reject:** if the entry is `[x]` (completed), print "TODO-N is completed. Completed entries are archived and cannot be revised."
   - **Reject:** if the entry only exists in TODOS-archive.md, print "TODO-N is archived. Archived entries cannot be revised."
3. **Scan for gaps:** Read the entry. Check each required and optional field:
   - **What:** present and non-empty?
   - **Why:** present and >= 10 characters?
   - **Decisions:** present and contains all sub-keys (Priority, Effort, Phase, Branch, Test Coverage, Security Review)?
   - **Pros/Cons/Context/Depends on/Assumptions:** present? (flag as optional gaps)
   - A field is "weak" if present but below minimums (e.g. Why < 10 chars). This is a new weakness heuristic specific to `--revise`. `--audit` only checks presence; `--revise` is stricter because it fills gaps, not just reports them.
4. **Auto-research scoped to gaps:** Read `sections/auto-research.md`. Run the research phase, then filter its full synthesis block to only the fields that are missing or weak. The existing entry's values for those fields are passed as additional context in the research prompt so derivations can improve on existing content rather than starting from a blank slate. This is a behavioral layering on top of auto-research.md, not a modification to the file itself — the scoping logic lives in `sections/revise.md`.
   - If auto-research fills nothing useful (budget cap hit with zero signals, or no keyword matches for any gap): present a clear "Auto-research found no signals for the missing fields. Please provide values manually:" and ask the user for each empty required field one at a time, then skip to Step 6.
5. **Present synthesis block:** Same format as --add's full synthesis block, but with all fields shown. Fields that were already good show with `(unchanged)` tag and their current value. Fields that were derived show with `Confidence:` tags. Fields the user provided manually show as `high` confidence.
6. **Single confirm/edit gate:** Consolidated into one gate (matching --add step 6). "Reply `confirm` to accept all, or list edits as `field: new value` — only the fields you mention change."
7. **Preview gate:** Show the full revised entry (not just deltas) so the user sees the complete before/after.
   ```
   ======== REVISION PREVIEW ========
   BEFORE:
   <current entry text>

   AFTER:
   <revised entry text>

   Proceed? [y / edit / cancel]
   ```
   - `y` → proceed to Step 8
   - `edit` → return to Step 6 (re-prompt for batch edits; does NOT re-trigger auto-research)
   - `cancel` → print "Revision discarded." and exit
8. **Write to TODOS.md:** Replace the entry at its original position (same order among entries). Entry boundaries are determined by the same heuristic as `--archive`: an entry starts with `- [ ]`, `- [→]`, `- [x]`, or `- [~]` and continues until the next entry header or end of file. Adding fields increases the entry's line count, shifting subsequent entries downward — line numbers change, but entry order is preserved.
9. **Confirm:** "✓ TODO-N revised. Updated fields: [list]."

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| All fields present and valid | Print "TODO-N has no missing or weak fields. Nothing to revise." and exit |
| User enters non-existent TODO-N | Print "TODO-N not found in TODOS.md." and exit |
| User enters archived TODO-N | Print "TODO-N is archived. Archived entries cannot be revised." and exit |
| User enters completed `[x]` TODO-N | Print "TODO-N is completed. Completed entries are archived and cannot be revised." and exit |
| Auto-research cap hit | Log which signals were skipped; treat remaining gaps as user questions |
| Auto-research fills nothing useful | Ask user for each missing required field one at a time |
| Write fails (file locked) | Print error; no partial write |
| Preview `edit` action | Return to Step 6 for batch re-edit; does NOT re-trigger auto-research |

## Example Subcommand Usage in Series

The `--revise` subcommand is designed to compose with other subcommands. Common workflows:

### Post-Convert Cleanup (most common)

After `--convert` migrates a header-based TODOS.md to canonical format, entries may have empty required fields:

```
1. todos-manager --convert          # Migrates format; reports missing fields
2. todos-manager --audit            # Confirms which entries have gaps
3. todos-manager --revise           # Fix TODO-9 (missing What, Why)
4. todos-manager --revise           # Fix TODO-11 (missing Decisions)
5. todos-manager --revise           # Fix TODO-12 (missing Why, Branch)
6. todos-manager --audit            # Verify all entries are now compliant
```

### Post-Audit Improvement

An audit finds weak or missing fields in entries that were originally well-formed:

```
1. todos-manager --audit            # Finds TODO-5 missing Branch, TODO-15 missing Decisions
2. todos-manager --revise           # Fix TODO-5 — auto-research fills Branch
3. todos-manager --revise           # Fix TODO-15 — auto-research fills Decisions
4. todos-manager --audit            # Clean slate
```

### Add-Then-Revise (iterative refinement)

A user adds a TODO quickly, then revises to add depth:

```
1. todos-manager --add              # Quick entry: TODO-19, minimal What/Why
2. (user reads the entry and notices gaps)
3. todos-manager --revise           # TODO-19 — fill in Pros, Cons, Context
4. todos-manager --list             # Verify the full list
```

### Full Project Lifecycle

```
1. todos-manager --init             # New project
2. todos-manager --add              # Add TODO-1, TODO-2, TODO-3
3. (work happens...)
4. todos-manager --archive          # TODO-1 is done
5. todos-manager --add              # TODO-4, TODO-5
6. todos-manager --audit            # TODO-5 has weak Why
7. todos-manager --revise           # Fix TODO-5
8. todos-manager --list             # Check project state
```

### One-Off Revision

A single entry needs updating (new branch name, priority change, etc.):

```
1. todos-manager --revise           # TODO-3 — update Phase from "Design" to "Development"
```

## Success Criteria

- One new section file, one SKILL.md update (Effort S).
- Reuses auto-research without reimplementing.

- Archived entries are never modified.
- Entry with no gaps gets a clear "nothing to revise" message.

### Outside Voice

**What's strong:** The design correctly identifies the audit-to-fix gap and reuses auto-research instead of reimplementing. The before/after preview gate is the right safety mechanism for destructive writes. The edge case table is comprehensive.

**What to watch:** The behavioral layering on auto-research is the most fragile part of the design — it's a thin abstraction but undocumented where the base pipeline lives. Write revise.md so the layering contract (what revise.md passes to auto-research.md and how it filters the synthesis) is explicit. That's the difference between "reuses" and "accidentally diverges."
