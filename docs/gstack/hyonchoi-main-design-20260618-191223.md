# Design: --verbose/--debug logging + counter recovery (TODO-13 + TODO-1)

## Problem Statement

Two small-but-useful gaps in developer experience:

1. **TODO-13:** Debugging pipeline issues requires manually digging through `.hermes/` state files. There is no way to surface internal state (selection agent input/output, circuit breaker internals, lock state) from the CLI without editing code.
2. **TODO-1:** When bootstrapping a project with hand-written `TODO-<n>` entries but no `.hermes/todo_id_counter` file, the todos-manager skill has no way to initialize the counter — ID collisions can occur on the first `todos-manager` add.

## What Makes This Cool

Not a "whoa" feature, but the kind of polish that makes the tool feel production-ready. `--debug` should be the first thing a developer types when something goes wrong, not something they need to implement by adding `.debug()` calls themselves.

## Constraints

- The existing `logging_setup.py` already has a `_TickFilter` and timed rotating file handler — the logging infrastructure is solid, just locked at INFO.
- The counter file uses atomic temp+rename writes with no lock (by design — see project learnings).
- No test coverage required for TODO-13 (per its Decisions). TODO-1 requires tests (per its Decisions).

## Premises

1. The existing `logging_setup.configure()` takes `log_path` and `retention_days` — adding a `level` parameter is the clean integration point.
2. `--verbose` maps to INFO (additional diagnostic detail), `--debug` maps to DEBUG (raw payloads, circuit breaker internals, lock state). Current code has mostly `.info()`/`.error()`/`.warning()` — `--debug` needs new `.debug()` calls to actually surface anything.
3. The counter recovery scan just needs a regex like `TODO-(\d+)` across TODOS.md, find the max, write to `.hermes/todo_id_counter` — no lock needed (per existing learning).
4. The todos-manager is a skill, but `--recover-counter` should also be available as a CLI subcommand for fast, bare-terminal usage.

## Approaches Considered

### Approach A: Level flags only (TODO-13)

3-5 line change. Add `--verbose` and `--debug` to the top-level parser, pass a `level` kwarg to `configure_logging()`. Existing INFO/DEBUG calls surface at those levels.

**Problem:** Current codebase has almost no `.debug()` calls. `--debug` would be a no-op without adding instrumentation lines.

### Approach B: Level flags + debug instrumentation (TODO-13, chosen)

Same as A, plus add targeted `log.debug()` lines at strategic spots: selection agent input/output, circuit breaker internals, lock acquisition/release, kanban registration results.

**Why chosen:** Without these lines, `--debug` surfaces nothing. The extra effort is worth it because the TODO-13 spec explicitly calls out "raw agent payloads, circuit breaker internals, lock file contents" as the value of `--debug`. The debug lines are gated by level — zero runtime cost when not using `--debug`.

### Approach A: Skill-only recovery (TODO-1)

Add `--recover-counter` as a todos-manager skill option. The skill scans TODOS.md, finds max ID, writes the counter file.

**Problem:** Requires skill invocation. A developer who just hit an ID collision won't want to invoke a skill — they want a fast command.

### Approach B: CLI subcommand + skill (TODO-1, chosen)

Add a `recover-counter` subcommand to `pipeline-watch`. The CLI reads TODOS.md, finds max ID, writes the counter file, prints the result. The todos-manager skill delegates to the same function.

**Why chosen:** Developer-friendly. `pipeline-watch recover-counter myproject` from any directory. Works outside Claude Code. The skill option calls the same function.

## Recommended Approach

**TODO-13:** Approach B — level flags + debug instrumentation. Add `--verbose`/`--debug` to the top-level argument parser. Pass `level` to `logging_setup.configure()`. Add `log.debug()` calls at ~6 strategic spots identified below.

**TODO-1:** Approach B — CLI subcommand + skill. Add `recover-counter` as a `pipeline-watch` subcommand with a `project` argument. The handler scans TODOS.md, extracts max `TODO-<n>`, writes `.hermes/todo_id_counter`, prints the result. The todos-manager skill calls the same handler.

## Resolved Questions

1. **--verbose log level:** Lower the default root log level from INFO to WARNING. Change existing `log.info()` calls that should always appear back to `log.warning()`. New `log.info()` calls at strategic spots (selection result summary, lock state transitions) become the --verbose surface. --debug at DEBUG level surfaces `log.debug()` calls (raw payloads, circuit breaker internals).
2. **recover-counter .hermes/ directory:** Create it if needed. The `recover_counter()` function should `Path(project_dir) / ".hermes" / "todo_id_counter"` — mkdir if missing. No `hermes init` prerequisite.
3. **Implementation separation:** Implement as two separate PRs/commits — TODO-13 in one, TODO-1 in another.

## Success Criteria

- `pipeline-watch tick --debug myproject` surfaces raw agent payloads, circuit breaker internals, and lock state in the log output.
- `pipeline-watch tick --verbose myproject` surfaces more detail (tick_id, lock state, selection results) than the default run.
- `pipeline-watch recover-counter myproject` scans TODOS.md, finds the max TODO-<n>, and writes it to `.hermes/todo_id_counter`.
- The todos-manager skill's `--recover-counter` option works the same way.
- All existing tests pass (281+ non-eval tests).

2. Lower the default root log level from INFO to WARNING. Demote existing `log.info()` calls that should always appear (errors, warnings) to `log.warning()`.
3. Modify `logging_setup.configure()` to accept an optional `level` parameter (default WARNING).
4. Pass the level from `cli.py`'s `main()` to `configure_logging()` based on the flags: `--verbose` → INFO, `--debug` → DEBUG, default → WARNING.
5. Add new `log.info()` calls at strategic spots (visible with --verbose):
   - Selection result summary (which TODO was picked, why)
   - Lock state transitions (acquired/released with lock file path)
6. Add `log.debug()` calls at these spots (visible with --debug):

   - `cli.py` — selection agent raw parsed output
   - `cli.py` — lock file path and holder PID on acquisition
   - `circuit.py` — circuit breaker state transitions (failure count, backoff state). **Note:** `circuit.py` has no logging infrastructure — add `import logging` and `log = logging.getLogger(__name__)`.
   - `kanban.py` — kanban registration payload
7. Verify with `uv run pipeline-watch tick --debug <project>` that the output includes the new debug lines.
8. Verify no existing tests fail due to new log output on stderr (debug calls at DEBUG level should not surface under the default WARNING level).
