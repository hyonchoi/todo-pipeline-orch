## Problem Statement

`pipeline-watch test` drives `run_harness()` (hermes_pipeline/harness.py:529), which
registers a mock pipeline's phases as kanban tasks and polls them to completion via
`_poll_kanban_phases()` (harness.py:251-379). Today that poll loop is silent: it computes
every phase status transition (running/done/failed) internally and forwards it only to
`monitor()`, which writes JSONL events to `events.jsonl` — nothing reaches the console.
A user running `test` sees nothing until the run finishes or the hard-coded `timeout`
kills it. Separately, `run_harness()` wraps the poll in `_run_with_timeout()`
(harness.py:494-526), which joins a daemon worker thread with a wall-clock `timeout` and
simply abandons the thread (never signals cancellation) if it fires — so a slow-but-healthy
pipeline can be killed mid-run with no way to see it coming.

## What Makes This Cool

Turns a black-box `test` run into something you can watch happen — phase-by-phase, in
real time, in the same terminal you started it from. No more staring at a spinner
wondering if it's stuck or just slow.

## Constraints

- Scoped to `hermes_pipeline/harness.py` only (the test-fixture poll loop) — not the
  production `pipeline-watch tick` path, which uses a different call chain
  (`cli.py` → `register_todo_phases` → kanban tasks spawn hermes agents outside this
  codebase) and has no equivalent silent-wait problem to begin with.
- Must not change `_run_with_timeout`'s threading/cancellation semantics — the daemon
  thread today is never cleanly joined on timeout; that gap already exists independent
  of this change and is out of scope to fix here.
- The `--timeout` CLI flag must remain functional as an opt-in safety fallback, just
  with a much larger default so it stops being the de-facto kill switch for normal runs.

## Premises

1. The `test` command is genuinely silent today during the poll loop — no console
   feedback exists until timeout/completion. **Agreed.**
2. Removing the wall-clock timeout ceiling (raising its default to effectively
   unbounded) is safe, because the daemon thread already isn't cleanly cancelled on
   timeout today — so a larger default doesn't remove any cleanup guarantee that
   currently exists. **Agreed.**
3. This is scoped to the harness/test-fixture poll loop only (`hermes_pipeline/harness.py`),
   not the production `pipeline-watch tick` path. **Agreed.**

## Approaches Considered

### Approach A: Minimal — inline print in the existing poll loop
Add `print()`/`log.info()` status-table output directly inside `_poll_kanban_phases()`'s
existing per-phase-transition branches (harness.py:325-358), reusing the `status_map`/
`previous_status` diff already computed each poll tick. Print a one-time initial status
table immediately after `register_todo_phases()` returns (harness.py:298); if
registration raises, the poll loop exits before printing anything (matches today's
behavior — no new failure path introduced). Change the `--timeout` CLI flag's default
(cli.py:321, currently `default=3600`) to a large value (e.g. 86400s) — `run_harness()`'s
`timeout` parameter (harness.py:535) already receives `args.timeout` verbatim from the
CLI (cli.py:1191), so no change to `run_harness()` or `_run_with_timeout` is needed.
- Effort: S
- Risk: Low
- Pros: smallest diff, touches only branches already computing transitions; no change
  to `_run_with_timeout`'s threading model — `--timeout` flag still works exactly as
  today, just defaults large.
- Cons: output is line-by-line transition logs rather than a redrawn live table; the
  one-time post-registration status table needs a separate print call outside the
  transition loop.

### Approach B: Ideal architecture — structured status renderer + config-driven timeout
Extract a `_render_status_table()` helper called once after registration and again on
every poll tick (diffed transitions only), backed by a small `PollStatus` dataclass.
Make timeout a documented config value; keep `_run_with_timeout` as an opt-in safety net.
- Effort: M
- Risk: Low-Med
- Pros: reusable renderer could serve other `pipeline-watch` subcommands later; clean
  separation between event data and display, easier to unit test.
- Cons: more surface area than the TODO's own Effort=`S` estimate; adds a new
  dataclass/module for what's currently ~30 lines of branch logic.

### Approach C: Lateral — reuse the existing HarnessMonitor/JSONL pipeline as the console sink
Instead of adding print statements inside `_poll_kanban_phases`, make `base_monitor`
(the `HarnessMonitor` already receiving every event) optionally tee to stdout, so console
output falls out of the existing event pipeline for free.
- Effort: S
- Risk: Low
- Pros: zero duplication between "what the monitor logs" and "what the console shows" —
  one event stream, one source of truth; automatically covers future event types.
- Cons: `_poll_kanban_phases` calls `monitor(...)` (the `_ConvergenceMonitor` wrapper),
  not `base_monitor` directly — a tee would need to sit inside a shared class used
  elsewhere in the codebase; the initial post-registration status table has no
  corresponding event to tee from, so it wouldn't fit this model cleanly.

## Recommended Approach

**Approach A.** The TODO's own Effort estimate is `S`; the code today already computes
every transition inline in `_poll_kanban_phases`; and both B and C add structure
(a reusable renderer, or a shared-class event tee) that no other consumer currently
needs. Minimal, surgical, matches scope.

## Success Criteria

- Running `pipeline-watch test` prints an initial phase status table immediately after
  `register_todo_phases()` returns (harness.py:298); if registration fails, the run
  exits before printing (no new failure path).
- Each phase transition (→ running, → done, → failed) is logged to the console as it's
  detected by the poll loop, not just written to `events.jsonl`.
- A healthy pipeline run is no longer killed by the default `--timeout`; the flag still
  works if a user explicitly sets a smaller value (verified via cli.py:1191, which
  already passes `args.timeout` through to `run_harness()` unchanged).

## Design decisions from review

- **Console output should use `log.info()`, matching the existing pattern.**
  Correction from an earlier draft of this review: `logging_setup.configure()`
  (`logging_setup.py:44-52`) attaches a `StreamHandler(sys.stderr)` to root
  *unconditionally* — not gated on `--verbose`/`--debug` — and defaults root's
  level to `logging.INFO`. So `log.info()` calls in `harness.py` already reach
  the console today with no flags needed; `-v`/`--debug` only raises root to
  `DEBUG` and additionally unmutes the separate `pipeline.verbose` logger
  (`WARNING` by default). Given this, the new transition/status-table output in
  `_poll_kanban_phases()` should use `log.info()` for consistency with the rest
  of the module (e.g. the existing `log.info("registering kanban phases for..."`
  call at harness.py:284) rather than introducing a second, inconsistent output
  channel via bare `print()`.

- **The raised `--timeout` default (86400s) stays a real safety net, not a formality.**
  `run_harness()`'s `finally: shutil.rmtree(temp_dir, ...)` only executes after
  `_run_with_timeout()`'s `worker.join(timeout)` returns — whether the daemon thread
  finished or is still hung. Today, a truly wedged poll (e.g. a kanban API call that
  blocks forever) leaves a stuck process for up to 1h before returning control (and
  even then, the daemon thread itself is abandoned, never cancelled — a pre-existing
  gap, per Premise 2, correctly out of scope here). Raising the default to 86400s
  raises that same worst case to 24h. This is an accepted, larger worst-case window —
  not something the plan should treat as disappearing because Premise 2 says no
  *cleanup guarantee* is lost. Keep the large default (it correctly stops being the
  de-facto kill switch for healthy long runs) but document in code/CHANGELOG that it
  is a genuine upper bound on a hang, sized generously, not a value chosen because it
  no longer matters.
