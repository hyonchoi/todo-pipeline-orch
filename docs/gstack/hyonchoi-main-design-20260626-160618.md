## Problem Statement

`todo-pipeline-orchestrator` runs as kanban-as-scheduler: each cron tick selects a TODO and
registers phases 2тАУ8 as Hermes kanban tasks via `register_todo_phases`
([kanban_tasks.py:61](hermes_pipeline/kanban_tasks.py)); Hermes goal agents execute them, Python
only registers and observes. In-flight projects are detected by `all_phases_complete`
([kanban_tasks.py:268](hermes_pipeline/kanban_tasks.py)) which holds the project while any task is
not in `COMPLETION_STATUSES = {done, failed}` ([kanban_tasks.py:33](hermes_pipeline/kanban_tasks.py)).

Two gaps:
1. **No ship gate.** Phase 8 (`finish-branch`) opens a PR and is terminal. Nothing merges it тАФ the
   PR sits on GitHub until a human notices. There is no first-class "awaiting approval" hold and no
   agent or command that performs the merge.
2. **No distinct "needs human input" state.** A task that needs intervention is indistinguishable
   from a clean failure, and nothing alerts the operator. (Deferred тАФ see Premises P2.)

This design adds an explicit ship gate, a deterministic `pipeline-watch approve` command that
merges on approval, and a single "ready to ship" operator alert.

## What Makes This Cool

The pipeline becomes a true unattended loop with exactly one human checkpoint at the only
irreversible step. Every tick the cron fires, projects advance on their own; the only thing that
waits for you is the merge-to-main, and when you approve, a deterministic command does the merge,
the version bump, and the changelog тАФ no agent improvising on `main`. The whole thing stays inside
the existing kanban state machine, so `hermes kanban list` remains the single source of truth for
"what is this project doing right now."

## Constraints

- Python 3.12+, uv-managed. Tests mock the `hermes` subprocess as existing tests do.
- Built on Hermes-native primitives only тАФ no custom column emulation.
- The active codepath is kanban-as-scheduler. The legacy in-process executor
  (`phases.py:_invoke_hermes` / `merge.py:run_phase9`) is NOT on this path and must not be assumed.
- Merging to `main` is a one-way door; the approval path must be deterministic and fully guarded.

## Premises

Five premises were challenged. Three were revised during the session.

**P1 тАФ Status model for the gate. REVISED, then reversed by tooling reality.**
Original plan: gate via `--initial-status blocked`. Challenge: Hermes has a first-class `review`
status, and overloading `blocked` (gate vs error) forces classification logic. The builder chose
`review` for the gate. **Verification then showed `review` is not CLI-settable** тАФ the transition
commands are `block`тЖТblocked, `unblock`тЖТready, `schedule`тЖТscheduled, `complete`тЖТdone,
`promote`тЖТready, `archive`, `claim`тЖТrunning; none sets `review`. `review` is an internal worker
status. **Resolution: keep `blocked` for the gate** (it is creatable via `--initial-status blocked`),
because P2 (below) removes the only reason `blocked` was ambiguous. Net: `blocked` is used for one
thing тАФ the gate тАФ so no classification logic is needed.

**P2 тАФ Error-blocked alerting (original Feature 2). DEFERRED.**
The plan's motivating trigger was "a goal agent that exhausts its turn budget тЖТ blocked." But
`hermes kanban block` is a manual command; nothing observed auto-transitions a stuck task to
`blocked` (turn exhaustion almost certainly yields `failed`, already handled). Feature 2 is deferred
until Step 0 confirms a real auto-`blocked` trigger exists. If only humans set `blocked`, Feature 2
is cut. **This deferral is what makes `blocked` unambiguous for the gate (see P1).**

**P3 тАФ Ship executor. REVISED.**
Original: unblock a 15-turn goal agent to merge. Challenge: merge-to-main is the one irreversible
action; an agent doing it non-deterministically is the wrong risk. **Resolution: `approve` performs
a deterministic Python merge** (`gh pr merge --squash` + version bump + CHANGELOG). Note the existing
`run_phase9`/`make_default_bump_fn` ([merge.py](hermes_pipeline/merge.py)) are NOT a clean reuse тАФ
they belong to the legacy path and do a local `git merge --no-ff`, not `gh pr merge`. The bump logic
in `make_default_bump_fn` is reusable; the merge mechanism is not.

**P4 тАФ Moving `terminal: true` from phase 8 to phase 9. FALSE BUT HARMLESS.**
`terminal` is only read by the legacy `_invoke_hermes` ([phases.py:203](hermes_pipeline/phases.py)).
The active path (`register_todo_phases`, `all_phases_complete`) ignores it. The real gate is
`blocked` тИЙ `COMPLETION_STATUSES`. Keep the flag move for legacy consistency, but it does no gating.

**P5 тАФ `approve` resolves the ship task by re-listing kanban. AGREED.**
Mirrors `get_todo_kanban_status`. Low risk.

## Cross-Model Perspective

Codex (gpt-5.5, read-only cold read) reviewed the reshaped design without seeing the conversation.

- **Steelman:** "`review` cleanly models human approval pending while `blocked` stays available for
  real execution trouble. Moving the merge into `pipeline-watch approve` puts the irreversible part
  under deterministic checks instead of a goal agent, and gating Feature 2 avoids building alerting
  around a state Hermes may not emit organically."
- **Biggest hole:** cron keeps ticking while approval is pending, so the gate must behave as an
  in-flight lock. (Mitigated: `blocked` тИЙ `COMPLETION_STATUSES`, so `all_phases_complete` returns
  False and `_tick_project` ([cli.py:855](hermes_pipeline/cli.py)) returns early before selection.
  The residual risk is **stale approval** тАФ approving a days-old PR against a moved base тАФ handled
  by the SHA guard below.)
- **Mechanism (c):** Codex recommended create-then-move-to-`review`. **This is not buildable** тАФ no
  CLI sets `review` (see P1). Adopting `blocked` instead.
- **Merge guards (adopted into Approach B):** record phase-8 PR head SHA; verify PR open + correct
  base + head SHA matches (or explicit `--force` re-approve) + checks green + clean tree + base
  fresh; idempotent merge (if already merged, verify base/head/SHA then just mark done); lock around
  `approve`. (Codex also suggested verifying the remote base contains the result post-merge тАФ **left
  out of the core: `gh pr merge` already returns a definitive result, and a remote re-check risks
  false negatives from eventual consistency. Track as a P1 hardening, not a blocker.**)

## Approaches Considered

### Approach A: Blocked-gate + deterministic approve (minimal)
`phase_9_ship` created `--initial-status blocked`; `approve` resolves it, runs `gh pr merge --squash`
+ bump + CHANGELOG, then `hermes kanban complete`. Basic guards only. Effort S, Risk Med. Thin
protection on a merge-to-main.

### Approach B: Blocked-gate + fully-guarded deterministic approve (RECOMMENDED)
Same gate as A, plus the full Codex guard set: record phase-8 PR head SHA at completion; verify PR
open + base + head SHA (or `--force`) + CI green + clean tree + base fresh; idempotent merge; approve
lock vs cron; refuse if phase 8 is not `done`. (Post-merge remote re-verification deferred to P1 тАФ
see Cross-Model Perspective.) Effort M, Risk Low.

### Approach C: Sentinel-hold gate, no phase-9 task (lateral)
Keep phase 8 terminal. On phase-8 done, `observe_outcomes` writes a `ready_for_ship` sentinel;
`_tick_project` skips selection while unconsumed; `approve` does the same guarded merge then clears
it. No kanban gate task. Effort M, Risk Med. Sidesteps kanban status tooling but creates two sources
of truth for "is the project free" and hides the hold from `hermes kanban list`.

## Recommended Approach

**Approach B.** Merging to `main` is the single irreversible action in the whole pipeline. The guards
(SHA-staleness, idempotency, approve lock) are not gold-plating тАФ they are the difference between a
gate you can trust unattended and one that quietly merges the wrong commit when you approve a
three-day-old PR. The gate uses `blocked` (CLI-creatable, unambiguous now that Feature 2 is
deferred), keeping kanban as the single source of truth.

### Shape of the work

  to phase 9 (legacy consistency only).

  in YAML. Optional-with-defaults is preferred. Do not add `--goal` flags for a gate task.
- **`hermes_pipeline/kanban_tasks.py`** тАФ when `phase.gate`, append `--initial-status blocked` to the

  **Persist the phase-8 PR head SHA in a sidecar** `.hermes/outcomes/<tick_id>-ship.json`, written
  when phase 8 is observed `done` (consistent with the existing `observe_outcomes` sidecar pattern;
  the gate task's body header is written at creation time, before the SHA exists, so a sidecar is the
  only workable location). `approve` reads this sidecar to verify the SHA.

  **Refuse with a clear error if phase 8 is not `done`** (the SHA-staleness guard depends on phase-8
  completion data; approving before phase 8 finishes is meaningless). Then run the guarded
  deterministic merge and `hermes kanban complete <ship_task>`. Add the one-time "ready to ship"
  Slack alert in `_tick_project`'s prior-tick branch, deduped by sidecar.

- **Deterministic merge helper** тАФ new `gh pr merge --squash` + bump (reuse `make_default_bump_fn`)

3. **Phase-8 PR head SHA persistence тАФ defaulted to a sidecar** `.hermes/outcomes/<tick_id>-ship.json`
   written when phase 8 is observed `done`. Suggested schema:
   `{"pr_head_sha": "<sha>", "pr_number": <n>, "base_branch": "<base>", "work_branch": "<branch>", "phase_8_task_id": "<id>"}`
   тАФ everything `approve` needs for its guards. Step 0 only needs to confirm phase-8 completion
   exposes the PR head SHA (via `gh pr view` on the recorded branch, or the agent's completion
   metadata).
4. **`approve` concurrency lock.** Reuse `TickLock` for the project, or a dedicated approve lock, so
   cron and a second operator cannot race the merge.
5. **Squash vs merge-commit** for `gh pr merge` тАФ confirm repo merge policy (squash assumed).

## Success Criteria

- `pipeline-watch tick <project>` registers 7 tasks; `phase_9_ship` shows `blocked` in
  `hermes kanban list --json`.
- While the gate is `blocked`, `all_phases_complete` returns False and the project is not re-selected.
- The tick after phases 2тАУ8 are `done` fires the "ready to ship" Slack alert exactly once (deduped).
- `pipeline-watch approve <project> --todo TODO-N` merges the PR deterministically only when all
  guards pass; refuses (clear message) when the PR head SHA has moved, CI is red, or the tree is
  dirty, unless `--force`.
- After a successful approve, the ship task is `done`, `all_phases_complete` is True, and the next
  tick selects new work for that project.
- Re-running `approve` on an already-merged PR is a no-op that marks the task done (idempotent).
- Unit/integration tests cover: gate gets `--initial-status blocked` and only the gate; `--parent`
  chain intact; `approve` resolves the right task id and merges only on passing guards; SHA mismatch
  is refused; idempotent re-approve; `all_phases_complete` False while gated, True once shipped;
  alert dedup across ticks.

### Section 4: Data Flow & Interaction Edge Cases

**Data flow: ship sidecar**
```
  Phase 8 done тЖТ observe_outcomes:
    INPUT: gh pr view output (branch name)
    VALIDATION: extract PR head SHA
    TRANSFORM: write .hermes/outcomes/<tick_id>-ship.json
    PERSIST: O_CREAT | O_APPEND, flock
    OUTPUT: sidecar JSON

  approve:
    INPUT: sidecar JSON
    VALIDATION: parse JSON, verify SHA
    TRANSFORM: gh pr view, compare SHA
    PERSIST: gh pr merge
    OUTPUT: merged PR + bump
```

**Shadow paths:**
- Nil sidecar (phase 8 didn't complete cleanly): `FileNotFoundError` тЖТ refuse with message
- Empty sidecar (0-byte file): `JSONDecodeError` тЖТ refuse with message
- Upstream error (gh fails mid-approve): `CalledProcessError` тЖТ refuse with message
- Partial write (sidecar write crashes): `O_CREAT | O_APPEND + flock` prevents partial reads
