## Vision

### 10x Check
Multi-repo pipeline orchestration — one Hermes Kanban board managing TODOs across multiple repos. The Python package becomes infrastructure, not a one-off.

### Platonic Ideal
The pipeline is a pure declaration of intent: read TODOS.md, emit Kanban tasks with dependencies, let Hermes dispatch execute each phase. The Python package has zero execution logic — no model calls, no subprocess management, no state machines. All policy lives in Hermes (model, auth, fallback, skills).

## Codex Review — Course Correction

Codex's outside-voice review identified 17 critical issues with Approach B:

1. Kanban is state storage, not a workflow engine — deleting the orchestrator loses real semantics
2. `hermes chat -q` is not a drop-in for `_anthropic_call()` — missing retries, model selection, error taxonomy
3. Migration deletes modules before proving Kanban can satisfy parity tests
4. Undefined handoff model — phase outputs lack durable paths
5. 5 accepted scope expansions add complexity before core is proven
6. Strategic miscalibration — optimizes for architectural purity over user-visible reliability
7. **Corrective**: build Hermes adapter under existing runner FIRST

**Resolution**: Phase A replaces two Anthropic call sites with `hermes chat -q`, keeps all existing modules. Phase B (Kanban-Native) is a future migration after Phase A is proven.
