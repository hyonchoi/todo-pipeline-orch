## Problem Statement

TODO-4 (massive integration test project) is P1 with zero implementation progress. The existing test suite has ~50 unit tests that cover individual modules in isolation, plus a narrow structural test harness for the todos-manager skill (`tests/skill-test-environment/`). There is no integration-level test that drives the full pipeline — Hermes decision agent + Kanban + Claude Code phases + git branching + state management — against mock project data. Debugging cross-system bugs requires manual setup each time.

TODO-19 is a partial implementation of TODO-4: a formalized, repeatable mock integration test harness that creates verifiable mock data and drives the entire pipeline from start to end.

## User Workflow

1. `hermes-pipeline test --fixture happy-path` — bootstraps a mock project in a temp dir, runs the real pipeline through all phases, produces a findings report.
2. Engineer reads the report, identifies a bug in code or prompts, applies a fix.
3. `hermes-pipeline test --fixture happy-path --loop` — re-creates the fixture from scratch, re-runs the pipeline, diffs the new report against the previous one to show what improved/regressed.
4. Repeat until the report shows clean end-to-end progression.

## Constraints

- **Real AI, real pipeline** — this is not a mock test. It invokes the actual `PipelineRunner`, real Hermes + Kanban + Claude Code subprocess calls against mock project data in a temp directory.
- **Local Hermes configuration assumed** — requires authenticated Hermes + Claude Code CLI on the machine running the harness.
- **Open source / research context** — built for self-use and sharing with colleagues, not a commercial product.
- **Single fixture first** — start with one formalized fixture that drives the happy path. Edge-case fixtures ("textures") come as extensions.
- **Repeatable setup, not deterministic output** — the fixture bootstrap is idempotent and deterministic. The AI produces different content on each run, so the harness verifies structural properties (phase ordering, kanban transitions, state file writes) rather than AI-generated content.
- **Config isolation** — set test model (`claude-haiku-4-5`) in fixture `.hermes/config.toml` and isolate Python-process state/locks via `PIPELINE_STATE_DIR`/`PIPELINE_LOCK_DIR` to prevent the harness from interacting with the user's real pipeline state. `HOME` is left untouched — the harness invokes real `hermes`/`claude` CLIs, which need the real `$HOME` to read auth credentials; overriding it would break the "real AI, real pipeline" constraint above for no isolation benefit (the fixture's own `.hermes/config.toml` and `PIPELINE_*` vars already isolate state).

## Premises

1. **Happy-path first, edge-cases later.** The first fixture drives a successful end-to-end pipeline run. Variations (broken configs, edge-case TODOS, etc.) are additions built on the same fixture format.
2. **Iterative debug cycle.** The harness loops by re-executing the pipeline after the user applies fixes, producing before/after findings reports. It does not autonomously write code fixes — it generates the report, the human applies the fix, the harness re-runs.
3. **Full real pipeline execution.** The test invokes actual `PipelineRunner`, real Hermes decision agent calls, real Kanban sync, real Claude Code phase dispatch. No mocking of AI components.
4. **Standalone CLI tool, not pytest.** Approach B — a standalone CLI command (`hermes-pipeline test`) that bootstraps mock projects, runs the pipeline, generates findings reports, and supports `--loop` for iterative debug cycles.

## Approaches Considered

### Approach A: Pytest Fixture Factory
Fastest to ship. Fits in existing `tests/` directory. Formal factory pattern for adding fixtures. No built-in looping or report generation. Tied to pytest for consumption.

### Approach B: Harness CLI Tool (Chosen)
Standalone CLI command (`hermes-pipeline test --fixture <name> --loop`). Bootstraps mock project in temp dir, drives real pipeline, generates structured findings report (JSON + markdown), supports `--loop` for iterative debug cycles. Report diffs between iterations show what improved/regressed.

### Approach C: Hybrid CLI + YAML Manifests + Pytest Gateway
Most extensible. YAML manifest schema for declarative fixtures. Dual consumption (CLI + pytest). Highest upfront design — needs a manifest schema before implementation.

## Recommended Approach

**Approach B: Harness CLI Tool.**

Start with Approach B because it validates the core risk — can we run the real pipeline on mock data and produce useful diagnostics? — without designing a manifest schema first. The fixture format emerges from the implementation, not a spec document.

**Decisions (committed defaults):**

| Decision | Default |
|----------|---------|
| Temp dir lifecycle | Clean up by default; `--keep` preserves for inspection |
| Fixture storage | Inline Python functions in v1; extract to files when pattern stabilizes |
| Assertion granularity | Minimum: phase ordering + kanban status match. Expand later. |
| Report output | File (JSON + markdown) + stdout summary |
| Loop diff | File-based: stores `<fixture>-report.<N>.json` per iteration, diffs adjacent pairs |
| Loop state | `--loop` re-creates fixture from scratch (idempotent bootstrap), does not reuse previous temp dir |

**Five deliverables (from TODO-19):**

1. **Setup script + mock project fixtures** — `create_mock_project(path, fixture_name)` factory function. Creates: `git init` + at least one initial commit on a named branch, preset `.hermes/config.toml`, TODOS.md with mock entries, minimal source files. The git state matches what `PipelineRunner` expects at startup (branch, commit history, no remote required).

2. **Pipeline execution through mock project** — Invokes the real `PipelineRunner` with the mock project directory as target. Real Hermes + Kanban + Claude Code subprocess calls. Per-phase timeout inherited from `phases.yaml` config; overall run timeout of 60 minutes (configurable via `--timeout`). 9 phases × 1800s default = 54min; 60min provides margin. If a phase hangs past its timeout, the harness records `timeout` as the phase outcome and continues to the next phase rather than aborting the entire run.

3. **Monitoring/verification of pipeline steps and kanban status** — Monitors via an optional `monitor` callback parameter added to `PipelineRunner.__init__`. When provided, `PipelineRunner` calls it on each phase transition (`phase_started`, `phase_completed`, `phase_failed`, `phase_timed_out`) and kanban sync (`set_active_task`, `update_phase`, `clear_active_task`). Callback is additive — existing `PipelineRunner` callers pass nothing and behavior is unchanged. The harness callback writes each event to a structured JSONL file: `{timestamp, event_type, phase_key, kanban_status, todo_id, duration_ms}`.

4. **Findings report generation** — Takes the JSONL event log, produces:
   - `report.json`: structured data with per-phase status, kanban state, duration, error message.
   - `report.md`: human-readable summary with phase progression table, mismatches, and "what to investigate" section.
   - Stdout: one-line summary ("7/9 phases passed; Phase 4 failed: hermes_call timeout").

5. **Loopable cycle** — `--loop` flag: re-creates fixture from scratch in a fresh temp dir, re-runs pipeline, produces `<fixture>-report.<N>.json`. After completion, diffs report.N against report.(N-1) as a per-phase comparison of (status, kanban_state, duration, error_message) with a summary of phases that changed status between runs. Output: "Phase 4: failed → passed (hermes_call returned 200). Phase 6: passed → failed (kanban sync timeout)."

**Timeout & failure recovery:**
- Per-phase: inherits `timeout` from `phases.yaml` (default 1800s).
- Overall run: 60-minute default, overridable via `--timeout` (9 phases × 1800s = 54min; 60min provides margin).
- If Hermes CLI is not found at startup: fail fast with `HermesDependencyError`-style message before creating the fixture.
- If a single phase times out: record as `timeout`, continue to next phase. The run is not aborted.
- Convergence detector: if 3+ consecutive phases fail with the same error class (e.g., `HermesCallError`), halt the run and report "convergence detector: repeated failures, likely misconfiguration." Configurable via `--convergence-threshold` (default: 3) — enough to distinguish transient API blips from genuine misconfiguration. Renamed from "circuit breaker" to avoid naming collision with existing `CircuitBreaker` in `circuit.py`.
- Overall timeout interaction: if the overall `--timeout` fires mid-phase, the harness records the in-progress phase as `timeout`, generates a partial report from the JSONL event log collected so far, and exits.

## Success Criteria

- A colleague can run `hermes-pipeline test --fixture happy-path` and see the full pipeline execute against mock data in under the pipeline's natural runtime (no harness overhead).
- The findings report identifies which phase failed and what kanban status was at the time of failure.
- Running `--loop` after a fix produces a diff showing what changed from the previous run.
- Adding a new fixture ("texture") requires creating one Python function — no changes to harness internals.

## Distribution Plan

Part of the `hermes_pipeline` package. New CLI subcommand registered in `cli.py`. Distributed as a package entrypoint (`hermes-pipeline test`). No separate binary or container — runs wherever `uv run hermes-pipeline` works. CI integration deferred until the harness proves useful locally.

## Cost Considerations

Each full pipeline run invokes real AI (Hermes + Claude Code) for every phase. The v1 fixture config pins `claude-haiku-4-5` in `.hermes/config.toml` for cost control; users may override to a more capable model for higher-fidelity testing. A single happy-path run: approximately 9 phases × Haiku API cost per phase — expect $1-3 per run. For `--loop` cycles, each iteration incurs the same cost. Measure actual cost after the first run and document in a follow-up edit.

**Rate limiting:** Anthropic API 429 responses trigger the existing retry mechanism in `hermes_adapter.py` (HERMES_RETRY_ATTEMPTS=2, HERMES_RETRY_DELAY=1s). If retries are exhausted, the phase records `rate_limited` as the failure reason and the run continues. Repeated rate-limit exhaustion across multiple phases triggers the convergence detector.

## Next Steps

1. **Define fixture factory interface** — `create_mock_project(path, fixture_name)` with git init + initial commit, TODOS.md, config, minimal source. Pin `claude-haiku-4-5` in fixture `.hermes/config.toml` for config isolation. `HOME` is not touched (real CLIs need real auth).
2. **Add optional `monitor` callback to `PipelineRunner.__init__`** — additive parameter; existing callers unaffected.
3. **Add `continue_on_failure` to `PipelineRunner`** — default `False` preserves current behavior; harness sets `True`. Bypass slug resolution by passing temp dir directly.
4. **Implement harness CLI subcommand** — `hermes-pipeline test --fixture <name>` with factory, runner invocation, JSONL event log.
5. **Build findings report generator** — JSONL → report.json + report.md + stdout summary. Normalize `error_class` to buckets (`"phase_failure"`, `"timeout"`, `"hermes_error"`, `"kanban_error"`, `"dependency_error"`). Bucket durations (`<1m`, `1-5m`, `5-15m`, `>15m`).
6. **Add `--loop` support** — fresh fixture bootstrap, file-based report diff.
7. **Implement convergence detector** — tracks 3+ consecutive same-class failures within a single run. `--convergence-threshold` (default: 3). Dependency-aware skip: if a phase fails, skip dependent phases.
8. **Create first edge-case fixture** — validate factory extensibility (e.g., malformed TODOS.md).

## Future Consideration

When the fixture factory pattern stabilizes (2-3 fixtures in use), consider extracting fixtures into YAML manifests (Approach C) for declarative, colleague-contributable fixture definitions. This is TODO-4 scope, not TODO-19.
