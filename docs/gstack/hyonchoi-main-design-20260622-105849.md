## Problem Statement

`pipeline-watch tick` requires a `project` argument — one Hermes cron entry per project. With multiple side projects this means N cron entries, N terminal outputs, and no single view of the portfolio. The kanban board is the per-project source of truth, but there's no control tower.

## What Makes This Cool

One cron fires, scans your projects directory, runs selection for every active project, and updates the kanban boards. You open the kanban view and see all your side projects moving forward on their own. The kanban board becomes the control tower — no dashboard needed.

## Constraints

- One TODO in flight per project (existing constraint)
- Single global lock for the scan — no two ticks overlap
- Hermes-only for LLM queries (TODO-6)
- Filesystem-based config — no database or network calls before selection

## Premises

1. One cron is better than N — one entry drives the whole pipeline.
2. The kanban board IS the control tower — no separate dashboard needed.
3. Filesystem markers (`project.toml` in `<project>/.hermes/`) are the right place for per-project config.
4. Selection runs in-process per-project within one tick (not spawned subprocesses).

## Architecture: Per-Project State

**Existing architecture has a critical limitation:** `state_dir` is a single global directory (`~/.hermes/`) shared across all projects. `current_tick_id.txt`, `outcomes/`, and `circuit.json` are global files — there's one circuit breaker and one prior-tick check for the entire pipeline.

With multiple projects, this doesn't work:
- Project A's tick_id would overwrite Project B's
- Project A's circuit breaker state would block Project B
- Outcomes from Project A would be observed against Project B

**Solution:** Per-project state directories under `<project>/.hermes/`. The global `state_dir` (`~/.hermes/`) remains for shared state (the global tick lock). Per-project functions (`_read_prior_tick_id`, `observe_outcomes`, `_make_circuit_breaker`) take a `state_dir` parameter and use `<project>/.hermes/` when called from the scan loop.

This is a refactor, not just additive — every function that reads/writes `current_tick_id.txt`, `outcomes/`, or `circuit.json` needs the per-project state_dir threading. The global `config.state_dir` remains for shared state (tick lock, global config overlay).

## Approaches Considered

### Approach A: Minimal Scan (Recommended)

Remove the `project` argument from `tick`, add a scan loop over `projects_dir`, and reuse the existing per-project tick logic inside the loop. Thin `project.toml` for `enabled` (archive without deleting TODOS.md) and `slack_channel` (per-project notification routing).

- **Effort:** M (~6 hours — per-project state_dir threading + project.toml + Slack channel resolution)
- **Risk:** Med (per-project state_dir threading is a refactor, not additive — runtime bugs possible if a call site is missed)
- **Pros:** Ships the scan loop with per-project filtering (archive without deleting TODOS.md) and Slack routing. Kanban board becomes the control tower without new UI code.
- **Cons:** Breaking change — `tick <project>` becomes `tick`. `project.toml` adds a new config surface.
- **Reuses:** `_cmd_tick` body, `TickLock`, `run_selection`, `register_todo_phases` — all existing.

### Approach B: Scan Only

Scan loop without `project.toml`. No per-project filtering, no Slack channel resolution.

- **Effort:** S (~4 hours)
- **Risk:** Low
- **Pros:** Ships fastest.
- **Cons:** Can't archive a project without deleting TODOS.md. All Slack notifications go to the global channel.

### Approach C: Two-Phase

Phase 1: Approach B (scan loop, no config). Phase 2: add `project.toml` and Slack as a separate TODO.

- **Effort:** S for Phase 1, S for Phase 2
- **Risk:** Low
- **Pros:** You get the control tower working this week. The second phase adds the "nice to haves" without blocking the core value.
- **Cons:** Two TODOs instead of one — more context switching. The scan loop may need refactoring to add project.toml later.

## Recommended Approach

**Approach A: Scan + project.toml.** The scan loop is the unlock — it changes how you operate every day. The `project.toml` with `enabled` and `slack_channel` is thin: `enabled` solves the archive problem (you can't delete TODOS.md to archive), and `slack_channel` is a 3-level fallback chain. Together they add ~30 lines of code. Include them rather than defer — they're the difference between "works" and "works in a real portfolio."

**On backward compatibility:** The `project` argument is removed. There is no backward-compatible path — the scan loop IS the tick. The existing single-project `tick myproject` callers update their cron entries to `tick`. This is a breaking change, not additive.

## Implementation Design

### 1. Per-Project State Directory

Each project gets `<project>/.hermes/` for its tick-specific state:

```
<project>/.hermes/
  current_tick_id.txt    # Per-project prior tick
  circuit.json           # Per-project circuit breaker
  outcomes/              # Per-project outcome files
```

The global `~/.hermes/` retains:
- `tick.lock` (global lock for the scan)
- `config.toml` (global selection/circuit breaker config)
- `kanban.db` (kanban state — already per-project via `board_slug`)

### 2. Scan Loop

Replace `tick <project>` with `tick` (no argument). The project slug is derived from the directory name (`d.name`).

```
tick:
  1. Acquire global TickLock
  2. Load global config overlay: _load_toml_overlay(config.state_dir, config)
  3. For each project in _discover_projects():
     a. project_state = project_dir / ".hermes"
     b. prior_tick_id = _read_prior_tick_id(project_state)
     c. If prior tick in-flight: skip this project
     d. mint per-project tick_id
     e. _make_circuit_breaker(project_state, cb_cfg, config.slack_channel) — per-project circuit breaker
     f. build SelectionContext, run_selection
     g. If picked=TODO-N: _persist_tick_id(project_state), register kanban
     h. If picked=None: observe circuit breaker, _persist_tick_id(project_state)
     i. If error: log.error("project {slug}: {error}"), skip to next project
  4. Release lock
```

### 3. Per-Project Config: `project.toml`

```toml
# <project>/.hermes/project.toml

[active]
enabled = true  # default if file missing; set false to archive

[notifications]
slack_channel = "project__my-slug"  # per-project alert channel
```

Location: `<project>/.hermes/project.toml` — lives next to the tick state files. Default (no file) is active — opt-out for archived projects.

**Channel resolution (priority):**
1. `project.toml`'s `slack_channel`
2. `PIPELINE_SLACK_CHANNEL` env var
3. `#alert` (hardcoded fallback)

### 4. Project Discovery

```python
def _discover_projects(config: Config) -> list[Path]:
    """Scan projects_dir for active projects with TODOS.md.

    The project slug is the directory name (d.name). Directories that fail
    _validate_project_slug are skipped with a warning. Projects with
    enabled=false in project.toml are skipped (archived).
    """
    projects = []
    for d in sorted(config.projects_dir.iterdir()):
        if not d.is_dir():
            continue
        slug = d.name
        if not _validate_project_slug(slug):
            log.warning("skipping invalid project slug: %r", slug)
            continue
        if not (d / "TODOS.md").exists():
            continue
        if not _is_enabled(d):
            continue
        projects.append(d)
    return projects
```

`_is_enabled(project_dir)` reads `<project>/.hermes/project.toml` if it exists, defaults to `True`.

### 5. Error Isolation

Per-project `try/except` in the scan loop. On error:
- Log at ERROR level with project name
- Continue to next project (don't abort the scan)

Matches the original design doc test case: "one project's TODOS.md malformed → isolated (logged), others proceed."

### 6. Slack Notifications

Per-project channel resolution (priority):
1. `project.toml`'s `slack_channel`
2. `PIPELINE_SLACK_CHANNEL` env var
3. `#alert` (hardcoded fallback)

Notification events:
- **New selection:** "Project X: picked TODO-N — '{title}'"
- **Circuit breaker trip:** "Project X: circuit breaker tripped"
- **Phase completion:** "Project X: TODO-N completed Phase {N}"

### 6. Backward Compatibility

Existing `pipeline-watch tick <project>` calls will fail (argument removed). Migration: update Hermes cron entries from `pipeline-watch tick myproject` to `pipeline-watch tick`. The state migration section handles moving `current_tick_id.txt`, `circuit.json`, and `outcomes/` from `~/.hermes/` to `<project>/.hermes/` — automatic on first scan.

**Known config change:** Users with projects outside `~/projects` need to set `PIPELINE_PROJECTS_DIR` (or `projects_dir` in config). This is documented in the Config class but needs to be called out in the tutorial.

## State Migration

Existing projects have `current_tick_id.txt`, `circuit.json`, and `outcomes/` in `~/.hermes/`. The scan loop needs these in `<project>/.hermes/`.

**Migration strategy:** On first `tick` run (no `project` arg), the scan detects if the global state files exist and migrates them:

```python
def _migrate_global_state(project_dir: Path, config: Config) -> None:
    """One-time: move per-project state from ~/.hermes/ to <project>/.hermes/."""
    project_state = project_dir / ".hermes"
    project_state.mkdir(exist_ok=True)

    for name in ("current_tick_id.txt", "circuit.json"):
        src = config.state_dir / name
        dst = project_state / name
        if src.exists() and not dst.exists():
            shutil.move(str(src), str(dst))
            log.info("migrated %s → %s", src, dst)

    # outcomes/ — move the directory, not individual files
    outcomes_src = config.state_dir / "outcomes"
    outcomes_dst = project_state / "outcomes"
    if outcomes_src.is_dir() and not outcomes_dst.exists():
        shutil.move(str(outcomes_src), str(outcomes_dst))
        log.info("migrated outcomes/ → %s", outcomes_dst)
```

This runs once per project on the first scan. If there are multiple projects sharing the same global state files, only the first project gets the migration (subsequent projects find the files already moved). This is acceptable because the first project is the only one that was actually running ticks — the global state files belong to the project that last wrote them.

## Success Criteria

- One cron entry (`pipeline-watch tick`) drives selection for all active projects
- A malformed TODOS.md in one project doesn't block selection in others
- Each project's kanban board reflects its current selection (the "control tower" is the set of kanban boards, not a new aggregation view — you iterate boards to see the portfolio)
