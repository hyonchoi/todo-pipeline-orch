## Problem Statement

The todos-manager skill (markdown-based, prompt-driven CLI for managing TODOS.md) lacks a regression test environment. When the skill is updated or used across different AI agents (Claude Code, Codex, Hermes), there's no automated way to verify that subcommands (`--add`, `--init`, `--convert`, `--audit`, `--archive`) produce correct output. The goal is a clean demo project with free-style TODOS.md fixtures, golden-file-based structural verification, AI-judged semantic validation, and agent-driven integration tests.

## What Makes This Cool

An AI agent driving a skill against a fixture, verifying correctness, and iterating until the implementation is sane — a closed-loop test harness for prompt-based tooling. This is meta-testing: agents testing agents.

## Constraints

- **In-repo fixture bundle** (Approach A chosen) — all test artifacts under `tests/skill-test-environment/`
- **Two-tier testing** — Python unit tests for schema logic + agent integration tests that spawn real agents
- **Hybrid verification** — regex-based structural assertions (file exists, entry count, ID sequence), AI judgment for semantic validation (markdown format, prompt quality)
- **General skill target** — the harness should drive any agent (Claude Code, Codex, Hermes) through a spawn-agnostic interface. The default implementation targets Claude Code; other agents add adapter modules.
- **Python/uv project** — leverage existing `hermes_pipeline/` test infrastructure

## Premises

1. The demo project with free-style TODOS.md is the test fixture — a clean, known-good project state that agents exercise the skill against.
2. The skill's deterministic logic (ID sequencing, entry parsing, archive moving, format validation) is implemented as Python modules that *duplicate* what the skill prompt instructs. The unit tests verify the Python; the agent integration tests verify the prompt. The Python serves as both test oracle and golden-file generator.
3. Agent integration tests are expensive, so they run on-demand (not per-commit) — the unit suite is the CI gate.
4. The test environment lives as a subdirectory within this repo (`tests/skill-test-environment/`) rather than a separate repo.

## Approaches Considered

### Approach A: In-Repo Fixture Bundle (Chosen)
All test artifacts under `tests/skill-test-environment/`. Demo project, golden files, unit tests, agent harness in one place. Fastest to ship, zero external dependencies.

### Approach B: Template-Generated Fixtures
YAML templates that programmatically generate demo project fixtures. Scales to 100+ edge cases but requires building a template renderer first. Can be layered on top of A later.

### Approach C: Standalone Demo Repo + External Harness
Separate git repo with real history, cloned into temp worktrees per test run. Most realistic but overkill — the skill operates on markdown, not git.

## Recommended Approach

**Approach A** with room to layer on B's templates later.

### Verification Module (`verify.py`)

The golden files are YAML descriptors — not markdown snapshots. Each declares assertions the verifier runs:

```yaml
# golden/add_happy_path.yaml
assertions:
  - file_exists: TODOS.md
  - regex_count:
      pattern: "^- \\[[ x~→\\]\\] TODO-\\d+:"
      count: 5  # original 4 + 1 new
  - regex_present: "^- \\[[ x~→]\\] TODO-5:"
  - preamble_present: true
  - max_id: 5
```

This avoids the brittleness of exact-string diffs while keeping assertions machine-readable. The golden file is the test case specification, not the expected output.

**Phase 1 — Structural (golden YAML assertions):**
- File existence checks (TODOS.md created, archive created)
- Regex-based entry count (pattern matches expected N lines)
- ID sequencing (max ID extracted via regex = expected value)
- Preamble presence (blockquote format rules exist)
- Status markers (correct markers for archived entries)

**Phase 2 — Semantic (AI judgment, integration tests only):**
- Only invoked for integration tests, never unit tests (unit tests are zero-token)
- Only invoked if Phase 1 passes (saves tokens)
- Spawns a lightweight agent subagent with the actual output + schema rules
- Agent judges: "Is this entry properly formatted per the todos-manager schema?"
- Returns pass/fail with one-line rationale

**Failure modes:**

| Scenario | Phase 1 behavior | Phase 2 behavior |
|----------|-----------------|------------------|
| Agent timeout | Test marked SKIP with timeout note | Not reached |
| Agent refusal | Test marked FAIL with refusal text | Not reached |
| Partial file write | Runs assertions against what exists — likely FAIL | Not reached |
| Missing file | `file_exists` assertion FAIL | Not reached |
| Phase 1 FAIL | Test FAIL with assertion diff | Skipped |
| Phase 1 PASS | — | Phase 2 runs |
| Phase 2 FAIL | — | Test FAIL with AI rationale |

### Agent Integration Test Pattern

The spawn mechanism uses a Python adapter interface. Each agent backend implements `spawn()` and captures file-system changes (not stdout, which is non-deterministic across agents):

```python
def test_agent_add_entry():
    """Spawn agent → run --add → verify structural + semantic output."""
    # 1. Copy demo-project to temp dir (fresh state, resolve symlinks)
    fixture = shutil.copytree("demo-project/", tmpdir / "project/", symlinks=True)

    # 2. Spawn agent with todos-manager skill + --add prompt
    #    Agent writes to fixture/TODOS.md; we verify the file system state
    result = agent_backend.spawn(
        skill_path=SKILL_PATH,
        prompt="Run todos-manager --add with title 'Add logging framework'...",
        project_dir=tmpdir / "project/",
        timeout=120,  # seconds
    )

    # 3. Phase 1: Structural verification (YAML assertions)
    actual_todos = read(tmpdir / "project/TODOS.md")
    verify_structural(actual_todos, "golden/add_happy_path.yaml")

    # 4. Phase 2: Semantic verification (AI judgment) — integration only
    if result.phase1_passed:
        verify_semantic(actual_todos, schema=TODOS_SCHEMA)
```

**Agent backend interface:**
```python
class AgentBackend(ABC):
    @abstractmethod
    def spawn(self, skill_path, prompt, project_dir, timeout) -> AgentResult: ...

class ClaudeCodeBackend(AgentBackend):
    """Spawn Claude Code via subprocess with --no-interactive and permission auto-allow."""
    ...

class CodexBackend(AgentBackend):
    """Spawn Codex via `codex exec` subprocess."""
    ...
```

The default backend is `ClaudeCodeBackend`. Other backends are behind `pytest.importorskip` — if an agent isn't installed, those tests skip gracefully.

**Pytest scoping:** The `tests/skill-test-environment/conftest.py` prefixes all fixtures with `skill_` (e.g., `skill_demo_dir`, `skill_golden_dir`) to avoid name collisions with the parent `tests/conftest.py`. Pytest auto-discovers parent confests regardless of `pytest_plugins`, so the naming namespace is the isolation boundary. The integration subdirectory has its own `conftest.py` for agent-specific fixtures.

**Permission handling:** Claude Code runs with `--permissions auto-allow` (or the equivalent settings.json) so the agent doesn't block on permission prompts. The test harness sets this via environment variables before spawning.

### Loop Development Pattern

For iterative refinement:
1. Run unit suite → catch structural bugs immediately
2. Run agent integration → catch prompt-following regressions
3. If AI judgment fails → examine the deviation → fix skill prompt or golden file
4. Re-run until all pass
5. When skill changes → re-run suite → update golden files if behavior intentionally changed

## Open Questions

1. **Golden file update workflow** — When skill behavior intentionally changes, what's the process for updating golden YAML assertions? Interactive diff? `--update-golden` flag that regenerates assertion counts from actual output?
2. **Non-deterministic LLM output** — Even structural output (field ordering, whitespace) may vary slightly across agent runs. The YAML assertions use regex counts, which tolerate reordering, but phrasing won't match. This is why Phase 2 (AI judgment) exists — it handles the variance structural checks can't. Accept that golden assertions cover structure, not content.
3. **Phase 2 scoping** — AI-judged semantic validation is ambitious for day one. Consider shipping Phase 1 (structural only) as v1, then adding Phase 2 as a follow-up. The verification module is designed to support both, but Phase 2 can be disabled with a pytest flag: `--no-semantic`.

## Success Criteria

- Unit test suite runs in <5 seconds with zero token cost
- Agent integration test runs one subcommand end-to-end and produces a pass/fail verdict
- At least one golden file per subcommand (`--add`, `--init`, `--convert`, `--audit`, `--archive`)
- Demo project TODOS.md covers all status markers and field combinations
- Running `uv run pytest tests/skill-test-environment/unit/` passes on a clean checkout

## Distribution Plan

No external distribution — this is internal test infrastructure. Checked into the repo under `tests/skill-test-environment/`. CI can run the unit suite; agent integration runs on demand via `uv run pytest tests/skill-test-environment/integration/`.
