## Problem Statement

The pipeline jumps from `phase_4_development` straight into `phase_6_1_cso` (security review). There's no code-quality review pass in between ‚Äî correctness bugs, reuse opportunities, and simplification cleanups only get caught at human PR review, or not at all. You want to trust the pipeline to ship code that's actually correct, not just secure.

## What Makes This Cool

The pipeline becomes a better engineer. A TODO goes in, code comes out with bugs caught and fixed before any human sees it. The review phase doesn't just report findings ‚Äî it fixes them autonomously. No human confirmation needed because there's no human at the terminal. You're building a pipeline that makes your pipeline better.

## Constraints

- **Unattended execution:** The phase runs via `hermes_agent_call` ‚Äî no human at the terminal. The `/review` skill asks for confirmation by default; the phase prompt must explicitly override: "apply fixes autonomously."
- **Hermes routing:** Per TODO-6, all LLM calls route through `hermes chat -q`. The phase prompt is passed to `hermes_agent_call`, which augments it with agent constraints and fires `hermes chat -q`.
- **Phase infrastructure:** Tools, turns, timeout come from `configs/phases.yaml`. Existing review phases (phase_2_autoplan, phase_6_1_cso) use 20 turns, 1800s timeout. The review-then-fix cycle needs more: **30 turns, 2400s timeout** ‚Äî review takes turns, fixing takes turns, testing takes turns.
- **Numbering:** Current sequence is 2 ‚Üí 3 ‚Üí 4 ‚Üí 6.1 ‚Üí 7 ‚Üí 8 ‚Üí 9. The new phase is `phase_5_review`, making it coherent.
- **Tools:** The phase needs `Edit` for applying fixes, `Bash` for running tests, `Read` for context. `tools: "Read,Write,Edit,Bash"`.

## Draft YAML Entry

```yaml
- phase_key: "phase_5_review"
  name: "Phase 5: Code Review"
  prompt: |
    Run the gstack /review skill on the current branch.

    AUTONOMOUS MODE ‚Äî there is no human at the terminal. Apply fixes without
    asking for confirmation. Do NOT pause for user input.

    Steps:
    1. Save a diff log of the current branch state to docs/pipeline/{todo_id}-review-diff.log
       (run: git diff main > docs/pipeline/{todo_id}-review-diff.log).
    2. Run /review to find correctness bugs and simplification cleanups.
    3. Apply all findings autonomously ‚Äî do NOT ask for confirmation.
    4. Run tests: uv run pytest. If tests pass, commit the fixes.
    5. If tests fail after applying fixes, revert the fix commit and commit a rollback note
       explaining which fixes caused test failures. The phase still completes.
    6. Save the review findings report to docs/pipeline/{todo_id}-review-findings.md.
    7. Commit all artifacts with message "review: address findings for {todo_id}".
  tools: "Read,Write,Edit,Bash"
  turns: 30
  timeout: 2400
```

## Premises

1. A new phase between `phase_4_development` and `phase_6_1_cso` that runs gstack `/review` skill with autonomous fix mode.
2. The phase prompt must explicitly instruct autonomous fixing ‚Äî `/review` asks for human confirmation by default; unattended phase needs "apply fixes autonomously." **A dry run must verify `/review` does not hang on confirmation inside the `hermes chat -q` subprocess BEFORE merge** (SESSION_KIND/SPAWNED_SESSION detection is unverified in that context).
3. **Rollback, tree-hygiene, test execution, and success verification are owned by `phases.py` code, not the prompt (Approach C).** On test failure or timeout the pipeline deterministically restores the pre-review worktree (`git reset --hard <PRE_HEAD>` + `git clean -fd`) and records a structured outcome. Success is a machine check (tests ran, tree clean, artifacts exist), not subprocess exit 0.
4. The phase fits as `phase_5_review`, keeping phase numbering coherent (2‚Üí3‚Üí4‚Üí5‚Üí6.1‚Üí7‚Üí8‚Üí9).
5. The review phase saves a pre-review AND post-review diff (using `git merge-base HEAD main` as the base), so CSO (phase 6.1) has accurate context on what review changed.
6. Tools are `Read,Edit,Bash` (Write dropped ‚Äî unused, expands blast radius in autonomous mode).

## Approaches Considered

### Approach A: Prompt Override with Inline Rollback (Chosen)
Add `phase_5_review` to `configs/phases.yaml`. The prompt invokes `/review`, saves a diff log first, applies fixes autonomously, runs `uv run pytest`, and reverts the fix commit if tests fail. One YAML entry. 30 turns, 2400s timeout. `tools: "Read,Write,Edit,Bash"`. The diff log gives CSO context on what changed.

### Approach B: Two-Phase Review (Find + Fix)
Split into `phase_5_review_find` and `phase_5_1_review_fix`. Find phase saves findings report and diff log, then HALTs. Fix phase applies fixes, runs tests, reverts on failure. More visibility, but more cost per TODO and more complex phase numbering.

### Approach C: External Rollback Script
Same as A, but rollback logic lives in a dedicated Python script (e.g., `hermes_pipeline/rollback.py`) instead of the prompt. The phase calls the script after `/review`. More robust rollback, but requires writing new code instead of a YAML-only change.

## Recommended Approach

**Approach C ‚Äî Code-Owned Rollback + Machine-Verifiable Success (chosen after /autoplan review, 2026-07-07).**

Both the Codex and independent Claude reviewers rejected Approach A: `phases.py:195-202` treats any non-zero return code ‚Äî including a timeout ‚Äî as a hard `RuntimeError` that stops the pipeline, so the prompt's promise that "the phase still completes after rollback" cannot hold. A timeout mid-edit leaves a dirty worktree for CSO. The YAML prompt has no machine-backed guarantee that tests ran, findings were saved, or the tree is clean.

Approach C moves control flow out of the prompt and into the pipeline. The prompt becomes instruction-only ("run /review with autonomous fix mode"). New Python code in the phase lifecycle owns:

```
phase_5_review lifecycle (code-owned):
  1. PRE:  capture HEAD sha + `git status --porcelain` (dirty-state snapshot)
           save pre-review diff:  git diff $(git merge-base HEAD main) > docs/pipeline/{todo_id}-pre-review.diff
  2. RUN:  hermes_agent_call(prompt="run /review with autonomous fix", tools="Read,Edit,Bash", 30t/2400s)
  3. POST (in phases.py, NOT the prompt):
           a. run `uv run pytest`  (captured, machine-checked)
           b. tests PASS  ‚Üí commit fixes; save post-review.diff + findings.md; outcome=review_clean
           c. tests FAIL  ‚Üí `git reset --hard <PRE_HEAD>` + `git clean -fd` (deterministic restore);
                            record outcome=review_reverted_test_failure (structured sidecar); phase still completes
           d. timeout / rc‚âÝ0 ‚Üí `git reset --hard <PRE_HEAD>` + `git clean -fd`;
                            outcome=review_timeout; phase FAILS cleanly (worktree restored, CSO sees clean tree)
  4. VERIFY: assert tree is clean (or restored), pytest actually executed, expected artifacts exist.
             Success == these checks pass, NOT just subprocess exit 0.
```

This is more code than one YAML line, but it makes the phase auditable and gives CSO a clean, known worktree in every branch. The prompt no longer carries rollback logic it can't reliably execute.

**Superseded ‚Äî Approach A (Minimal Prompt Override):** One YAML entry; the prompt instructs `/review` to apply fixes, run tests, and revert on failure. Rejected because the success/rollback contract lives in prose with no infrastructure backing (see review report below).

## Open Questions

1. **Autonomous mode:** Does `/review` proceed without prompting when the prompt says "AUTONOMOUS MODE"? If the skill still asks for confirmation, the phase hangs. **Safety net:** the phase has a hard timeout (2400s) and a turn limit (30). If it hangs, the timeout kills it. Before shipping, validate with a dry run on a feature branch.
2. **Effort level:** The `/review` skill handles effort internally. Revisit after 3-5 TODOs have run through the phase. If the review misses bugs consistently, adjust the prompt to request deeper analysis.
3. **Commit strategy:** One commit for the review phase ‚Äî fix changes, findings report, and diff log together with message `review: address findings for {todo_id}`. Matches the pattern used by existing phases (autoplan, cso, document_release).
