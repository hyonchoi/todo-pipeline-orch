## Problem Statement

The pipeline jumps from `phase_4_development` straight into `phase_6_1_cso` (security review). There's no code-quality review pass in between — correctness bugs, reuse opportunities, and simplification cleanups only get caught at human PR review, or not at all. You want to trust the pipeline to ship code that's actually correct, not just secure.

## What Makes This Cool

The pipeline becomes a better engineer. A TODO goes in, code comes out with bugs caught and fixed before any human sees it. The review phase doesn't just report findings — it fixes them autonomously. No human confirmation needed because there's no human at the terminal. You're building a pipeline that makes your pipeline better.

## Constraints

- **Unattended execution:** The phase runs via `hermes_agent_call` — no human at the terminal. The `/review` skill asks for confirmation by default; the phase prompt must explicitly override: "apply fixes autonomously."
- **Hermes routing:** Per TODO-6, all LLM calls route through `hermes chat -q`. The phase prompt is passed to `hermes_agent_call`, which augments it with agent constraints and fires `hermes chat -q`.
- **Phase infrastructure:** Tools, turns, timeout come from `configs/phases.yaml`. Existing review phases (phase_2_autoplan, phase_6_1_cso) use 20 turns, 1800s timeout. The review-then-fix cycle needs more: **30 turns, 2400s timeout** — review takes turns, fixing takes turns, testing takes turns.
- **Numbering:** Current sequence is 2 → 3 → 4 → 6.1 → 7 → 8 → 9. The new phase is `phase_5_review`, making it coherent.
- **Tools:** The phase needs `Edit` for applying fixes, `Bash` for running tests, `Read` for context. `tools: "Read,Write,Edit,Bash"`.

## Draft YAML Entry

```yaml
- phase_key: "phase_5_review"
  name: "Phase 5: Code Review"
  prompt: |
    Run the gstack /review skill on the current branch.

    AUTONOMOUS MODE — there is no human at the terminal. Apply fixes without
    asking for confirmation. Do NOT pause for user input.

    Steps:
    1. Save a diff log of the current branch state to docs/pipeline/{todo_id}-review-diff.log
       (run: git diff main > docs/pipeline/{todo_id}-review-diff.log).
    2. Run /review to find correctness bugs and simplification cleanups.
    3. Apply all findings autonomously — do NOT ask for confirmation.
    4. Run tests: uv run pytest. If tests pass, commit the fixes.
    5. If tests fail after applying fixes, revert the fix commit and commit a rollback note
       explaining which fixes caused test failures. The phase still completes.
    6. Save the review findings report to docs/pipeline/{todo_id}-review-findings.md.
    7. Commit all artifacts with message "review: address findings for {todo_id}".
  tools: "Read,Write,Edit,Bash"
  turns: 30
  timeout: 2400
```

## Premises

1. A new phase between `phase_4_development` and `phase_6_1_cso` that runs gstack `/review` skill with autonomous fix mode.
2. The phase prompt must explicitly instruct autonomous fixing — `/review` asks for human confirmation by default; unattended phase needs "apply fixes autonomously."
3. The review phase runs tests (`uv run pytest`) after applying fixes and reverts the fix commit on test failure — preventing silent degradation.
4. The phase fits as `phase_5_review`, keeping phase numbering coherent (2→3→4→5→6.1→7→8→9).
5. The review phase saves a diff log before applying fixes, so CSO (phase 6.1) has context on what changed and why.

## Approaches Considered

### Approach A: Prompt Override with Inline Rollback (Chosen)
Add `phase_5_review` to `configs/phases.yaml`. The prompt invokes `/review`, saves a diff log first, applies fixes autonomously, runs `uv run pytest`, and reverts the fix commit if tests fail. One YAML entry. 30 turns, 2400s timeout. `tools: "Read,Write,Edit,Bash"`. The diff log gives CSO context on what changed.

### Approach B: Two-Phase Review (Find + Fix)
Split into `phase_5_review_find` and `phase_5_1_review_fix`. Find phase saves findings report and diff log, then HALTs. Fix phase applies fixes, runs tests, reverts on failure. More visibility, but more cost per TODO and more complex phase numbering.

### Approach C: External Rollback Script
Same as A, but rollback logic lives in a dedicated Python script (e.g., `hermes_pipeline/rollback.py`) instead of the prompt. The phase calls the script after `/review`. More robust rollback, but requires writing new code instead of a YAML-only change.

## Recommended Approach

**Approach A — Minimal Prompt Override.** The simplest change that works. One new YAML entry in `configs/phases.yaml`. A well-crafted prompt instructs `/review` to apply fixes autonomously and run tests after. If the skill hangs waiting for confirmation, that's immediate feedback and we iterate to Approach B.

## Open Questions

1. **Autonomous mode:** Does `/review` proceed without prompting when the prompt says "AUTONOMOUS MODE"? If the skill still asks for confirmation, the phase hangs. **Safety net:** the phase has a hard timeout (2400s) and a turn limit (30). If it hangs, the timeout kills it. Before shipping, validate with a dry run on a feature branch.
2. **Effort level:** The `/review` skill handles effort internally. Revisit after 3-5 TODOs have run through the phase. If the review misses bugs consistently, adjust the prompt to request deeper analysis.
3. **Commit strategy:** One commit for the review phase — fix changes, findings report, and diff log together with message `review: address findings for {todo_id}`. Matches the pattern used by existing phases (autoplan, cso, document_release).

## Success Criteria

- A new `phase_5_review` entry exists in `configs/phases.yaml` with correct position in the sequence.
- The phase prompt instructs `/review` to run with autonomous fixing.
- The phase runs tests after applying fixes.
- Phase numbering is coherent: 2 → 3 → 4 → 5 → 6.1 → 7 → 8 → 9.
- The phase integrates with existing infrastructure (tools, turns, timeout, Hermes adapter).
