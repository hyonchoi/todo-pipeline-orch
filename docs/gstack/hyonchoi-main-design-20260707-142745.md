## Problem Statement

The autonomous pipeline runs TODOs from `phase_2_autoplan` through `phase_9_ship` with only one human touchpoint: the ship gate. After Autoplan produces a plan, the pipeline proceeds straight into Development (phase 4) with no validation checkpoint. The builder currently watches pipeline runs and mentally validates plans — tacit judgment trapped in their head.

## Demand Evidence

**Hypothetical / thought experiment.** No external customers yet. The builder is the sole user. The evidence is internal: repeated manual intervention to catch plan quality issues before they waste development cycles. "I've had to stop the pipeline mid-run because Autoplan produced a plan that missed a critical dependency or had wrong phase sequencing."

## Status Quo

**Manual / ad-hoc workaround.** The builder watches pipeline runs (via `pipeline-watch tick` or cron) and mentally validates each plan. When a bad plan slips through, they either:
- Kill the pipeline run and restart (wasting the tick)
- Let it run to ship gate, then reject there (wasting all downstream phases)
- Fix the plan by editing TODOS.md mid-run and re-triggering

This costs ~15-30 min per bad plan caught, plus cognitive load of constant monitoring.

## Target User & Narrowest Wedge

**Just me — I'm the user.** A solo builder running an autonomous pipeline for personal/internal projects.

**Narrowest wedge:** A single decision sheet produced by Autoplan, one async approval via Kanban board (same UX as ship gate), one CLI command (`approve-plan`) to unblock. No multi-reviewer, no UI, no LLM scoring — just the artifact and the gate.

## Constraints

- Must reuse existing ship-gate machinery (`approve_lock`, `ShipSidecar`, runner checkpoint logic, Kanban gate registration)
- Must compose with existing: runner checkpoint logic, Kanban status transitions, `run_selection` agent
- Plan Gate sits between `phase_2_autoplan` and `phase_3_writing_plan`
- Decision sheet must be stable before phase 3 starts (phase 3+ append to ledger)
- Decision sheet directory: `.hermes/decisions/` (plural, matches existing codebase convention used by `hermes_pipeline/decision/store.py` and `HermesSelectionDecision`)
- Gate timeout: runner re-polls each tick cycle; no operator approval → TODO remains blocked indefinitely (operator must act)
- Concurrent approval protection: `approve_lock(*, state_dir: Path, gate_key: str, tick_id: str)` uses fcntl flock on `.hermes/decisions/<tick_id>-<gate_key>.lock`. The tick-scoped path allows parallel approvals for different ticks of the same gate type. Second caller gets "gate already being processed" error.
- **Runner pause mechanism**: The `PipelineRunner.run()` loop (in `runner.py`) is modified to check `phase.gate` before calling `run_phase_fn`. For gated phases:
  1. Runner calls `check_gate_status(state_dir=self.state_dir, gate_key=phase.phase_key, tick_id=self.tick_id)` to get a `GateStatus` enum: `BLOCKED`, `READY`, `RUNNING`, `FAILED`, `UNKNOWN`.
  2. If status is `BLOCKED` or `READY`, runner **does not sleep in-place** (that would block the tick). Instead, runner writes a `GateWait` marker to `.hermes/state/<tick_id>-gate-wait.json` containing `{todo_id, gate_key, timestamp}`. The runner returns `False` to its caller (the tick dispatcher in `watcher.py`). The `False` return signals "this TODO did not complete" — the dispatcher sees the `GateWait` marker and skips re-selecting this TODO for the next Autoplan run. The dispatcher's tick interval (typically 1-5 min) provides natural backoff.
  3. On the next tick cycle, runner re-enters the gated phase. If status becomes `RUNNING`, runner short-circuits the phase by returning `rc=0` directly in Python (skipping `run_phase_fn` entirely). This avoids calling Hermes with 0 turns.
  4. If status is `FAILED` (rejected), runner exits the TODO with failure and records the rejection in the outcome.

- **Gate readiness alert (dispatcher)**: `maybe_gate_ready(*, project_dir, project_slug, prior_tick_id, state_dir, slack_channel, gate_key)` runs once per tick by the dispatcher. It calls `check_gate_status` for pending gates; if status becomes `RUNNING`, it removes the GateWait marker and optionally sends a Slack alert. This follows the `maybe_ship_ready()` pattern.

- **Recovery from rejected state**: Rejected TODOs have Kanban status `failed` and a rejection sidecar. Operator can re-trigger by running `pipeline-watch tick` — the selection agent will re-evaluate the TODO (same tick_id or new) and re-run Autoplan, producing a fresh decision sheet.

- **Gate naming contract**: The `gate_key` parameter must match the phase `phase_key` field in `phases.yaml` (e.g., `phase_2b_plan_gate`, `phase_9_ship`). Lock files follow the pattern `.hermes/decisions/<tick_id>-<gate_key>.lock`. The generalized `approve_lock` in `gates.py` takes `gate_key` and uses this pattern — plan gate and ship gate use different lock files for the same `tick_id`, avoiding collision.

- **todo_id type canonicalization**: The codebase uses `todo_id` as string (e.g., `"TODO-5"`) in `phases.py`, `runner.py`, and `watcher.py`. The `ShipSidecar` and decision sheet use integer (5) for storage. The `gates.py` module provides `todo_int(todo_id: str) -> int` and `todo_str(todo_id: int) -> str` helpers. Conversion happens at the module boundary: decision sheet loads int, converts to string for runner; runner calls gates with string, gates converts to int for artifact paths.

### Terminology

- **Decision Sheet**: The artifact produced by Autoplan — a JSON file (`.hermes/decisions/<tick_id>-plan.json`) and markdown summary (`docs/pipeline/<TODO-N>-decisions.md`) containing the plan's questions, options, and recommended answers. Includes `schema_version: "1.0"`. **Note:** `.hermes/decisions/` is the data directory (created at runtime), distinct from the Python package `hermes_pipeline/decision/`.
- **Decision Ledger**: The running log of decisions made during implementation (phases 3-7). Starts as a copy of the approved Decision Sheet with `answer` fields filled. Appended to as JSONL in `.hermes/decisions/<tick_id>-ledger.jsonl` by each phase. **Deferred to follow-up TODO** — the ledger format is defined but integration with phases 3-7 is deferred. Ship-gate drift detection reads the ledger when implemented.
- **Override**: In the plan-gate context, "override" means "approve with annotated corrections." The operator provides corrected answers for specific `question_id`s via `--override q1=B`; these corrections are written into the Decision Sheet's `answer` field (original `recommendation` preserved) and carried into the Ledger with an `override_from` field capturing the original recommendation. The override is visible at ship time as "decision X overridden from A to B" via ledger diff. **Operator can only override existing question answers, not add new questions or change recommendations.**

## Phase Definition (phases.yaml)

The Plan Gate is a new phase inserted between `phase_2_autoplan` and `phase_3_writing_plan`:

```yaml
- phase_key: phase_2b_plan_gate
  name: "Plan Gate — Human Review"
  gate: true
  prompt: "Autoplan has produced a decision sheet awaiting human review. The decision sheet is at .hermes/decisions/<tick_id>-plan.json. Review the decisions, then run: uv run approve-plan <tick_id> --approve|--reject [--override q1=B] [--reason '...']. The gate will unblock when the Kanban task status becomes 'running'."
  tools: []
  turns: 0
```

**Position in phase list:** After `phase_2_autoplan`, before `phase_3_writing_plan`. The `gate: true` flag signals the runner to pause before executing this phase.

Autoplan (gstack skill `~/.local/share/gstack/autoplan/SKILL.md`) currently produces:
- A plan document at `docs/pipeline/{todo_id}-plan.md` (markdown, human-readable)
- No machine-readable decision artifact

The decision sheet adds a machine-readable artifact: `.hermes/decisions/<tick_id>-plan.json` with:
```json
{
  "schema_version": "1.0",
  "todo_id": 5,
  "tick_id": "01HX...",
  "questions": [
    {
      "question_id": "q1",
      "classification": "taste|premise|user-challenge|mechanical",
      "prompt": "Question text",
      "options": [{"label": "A", "description": "..."}, {"label": "B", "description": "..."}],
      "recommendation": "A",
      "rationale": "Why A was chosen by the 6 principles",
      "answer": null
    }
  ]
}
```

**Note:** `todo_id` is stored as integer (5) to match `PipelineRunner` internal type. The markdown summary at `docs/pipeline/<TODO-N>-decisions.md` displays it as "TODO-5" for readability.

The **stub generator** in `hermes_pipeline/gates.py` maps from Autoplan's existing markdown plan to this schema by:
- Extracting a "## Decisions" section from `{todo_id}-plan.md` with a predictable format:
  ```markdown
  ## Decisions
  
  ### Q1: Question text
  **Options:** A) ... | B) ...
  **Recommendation:** A
  **Rationale:** ...
  ```
- Parsing each decision as a question with recommendation
- Emitting a valid decision sheet with `answer: null` (unapproved)
- This allows gate machinery development before the Autoplan skill is modified
- **This stub is throwaway development infrastructure** — it parses the exact format above and will be replaced when Autoplan is modified to emit the JSON directly.

### Markdown Decision Summary Format

The human-readable summary at `docs/pipeline/<TODO-N>-decisions.md`:
```markdown
# Decision Sheet: TODO-5

**Tick ID:** 01HX...
**Schema Version:** 1.0
**Generated:** 2026-07-07T14:30:00Z

## Questions

### q1: [Question text]
**Classification:** taste
**Recommendation:** A
**Rationale:** Why A was chosen by the 6 principles

**Options:**
- **A:** Description of option A
- **B:** Description of option B

**Answer:** [PENDING]
```

**Schema validation:** `hermes_pipeline/decision/schema.py` adds a new frozen dataclass `DecisionSheet` (matching the existing `HermesSelectionDecision` pattern, avoiding Pydantic dependency). The model fields match the JSON schema above with these validation rules applied at load time:
- `questions`: list with at least 1 item
- Each question: `options` must have ≥2 items; `recommendation` must match one option label; `classification` must be one of `taste|premise|user-challenge|mechanical`
- `schema_version` must be `"1.0"`
- `todo_id` must be positive int
The stub generator and `approve-plan` CLI both use this model — a malformed decision sheet fails fast at load time.

## `approve-plan` CLI Specification

Module: `hermes_pipeline/cli/approve_plan.py` (new file, registered in `cli.py` via Typer)
Command: `uv run approve-plan <tick_id> --approve|--reject [--override q1=B] [--reason '...']`

Behavior:
1. Load decision sheet from `.hermes/decisions/<tick_id>-plan.json` (validates via `DecisionSheet` model)
2. Acquire `approve_lock(*, state_dir=state_dir, gate_key="phase_2b_plan_gate", tick_id=tick_id)` (fcntl, fails if lock held)
3. If `--approve`:
   - Fill `answer` field for each question: use override value if provided, else use `recommendation`
   - Write updated decision sheet back (answers now filled)
   - Mark Kanban task status `running` in `.hermes/state/<tick_id>.json`
   - Remove GateWait marker if present
4. If `--reject`:
   - Write rejection sidecar `.hermes/decisions/<tick_id>-rejected.json` with `{reason, timestamp, rejection_count}`
   - Mark Kanban task status `failed` in `.hermes/state/<tick_id>.json`
5. Release lock.

Override edge cases:
- `--override q1=X` where `q1` not in decision sheet → error, no partial writes
- Duplicate `--override q1=A --override q1=B` → error (last one wins is ambiguous)
- Override value must match an existing option label → error if not

## Gate Status Check Specification

`check_gate_status(*, state_dir: Path, gate_key: str, tick_id: str) -> GateStatus` reads the Kanban task status from `.hermes/state/<tick_id>.json` and returns:

| GateStatus | Condition |
|------------|-----------|
| `BLOCKED` | Kanban task exists with `status: "blocked"` |
| `READY` | Kanban task exists with `status: "ready"` (operator has reviewed but not approved) |
| `RUNNING` | Kanban task exists with `status: "running"` (approved via `approve-plan`) |
| `FAILED` | Rejection sidecar exists at `.hermes/decisions/<tick_id>-rejected.json` |
| `UNKNOWN` | No Kanban task found for this `gate_key` and `tick_id` |

This function is pure — no side effects. Called by both runner (to gate execution) and dispatcher (to detect ready gates).

## Kanban Task Creation

The plan-gate Kanban task is created by `register_todo_phases()` in `kanban_tasks.py` when the TODO is first selected, alongside all other phase tasks. The phase definition YAML with `gate: true` causes the task to be registered with initial status `blocked`. The selection agent does not create tasks — it only selects which TODO to run next based on Kanban state. The task creation is a one-time setup per TODO at selection time.

- `register_todo_phases(project_slug, todo_id, phases)` iterates over the phase list from `phases.yaml`
- For `phase_2b_plan_gate` with `gate: true`, it calls `hermes kanban create --status blocked --phase phase_2b_plan_gate --parent phase_2_autoplan` (dependency on autoplan phase)
- The runner reads the task status from `.hermes/state/<tick_id>.json` on each tick cycle

## GateWait Marker Schema

File: `.hermes/state/<tick_id>-gate-wait.json`
```json
{
  "todo_id": 5,
  "gate_key": "phase_2b_plan_gate",
  "timestamp": "2026-07-07T14:30:00Z",
  "tick_id": "01HX..."
}
```
The dispatcher (in `watcher.py` tick loop) reads this marker before calling `PipelineRunner.run()`. If present for the current `tick_id` and `todo_id`, it skips re-selection of this TODO for Autoplan and proceeds to the runner for gate re-check.

## Runner Gate Logic Pseudocode

```python
# In PipelineRunner.run()
for phase_index, phase in enumerate(self.phases):
    if phase.gate:
        gate_key = phase.phase_key
        status = check_gate_status(state_dir=self.state_dir, gate_key=gate_key, tick_id=self.tick_id)
        
        if status in (GateStatus.BLOCKED, GateStatus.READY):
            # Write GateWait marker
            self._write_gate_wait_marker(gate_key)
            return False  # Signal to dispatcher: TODO not complete
            
        elif status == GateStatus.RUNNING:
            # Short-circuit: gate approved, no Hermes call needed
            # Must mark phase done for checkpoint consistency
            self.state.mark_phase_done(self.todo_id, gate_key, phase_index)
            continue  # Next phase
            
        elif status == GateStatus.FAILED:
            self._record_rejection(gate_key)
            return False  # Signal to dispatcher: TODO rejected
    
    # Normal phase execution for non-gated or approved-gated phases
    rc = self.run_phase_fn(phase)
    if rc != 0:
        return False
return True
```

## Approaches Considered

### Approach A: Minimal Viable — Generic Gate Machinery (Chosen)

**Summary:** Generalize the existing Ship Gate pattern into a reusable `gates.py` module. Add `phase_2b_plan_gate` as a blocked Kanban task. Autoplan writes a decision sheet; the `approve-plan` CLI mirrors `approve-ship` with overrides/reject. Phase 3-7 append to a decision ledger; ship-gate Slack alert may include ledger diff in a follow-up.

**Effort:** M | **Risk:** Medium

**Pros:**
- Single source of truth for gate logic (no duplication)
- Ship-gate tests pass after generalization (regression guard)
- Decision sheet + ledger gives full audit trail
- Same async Kanban UX as ship gate — operator approves from board

**Cons:**
- Requires refactoring `ship.py` → `gates.py` (touching working code)
- Decision sheet schema must be stable before phase 3 starts
- `ship.py` refactoring must carefully separate generic gate functions (`write_sidecar`, `read_sidecar`, `approve_lock`, `GateSidecar`, `write_gate_sidecar`, `read_gate_sidecar`, `maybe_gate_ready`) from ship-specific logic (`gh_pr_merge_squash`, `bump_in_pr`, `approve_ship`, `ShipSidecar`)

**Refactoring strategy:**
1. Create `hermes_pipeline/gates.py` with generic `GateSidecar` dataclass (generalizes `ShipSidecar`)
2. Move `write_sidecar`, `read_sidecar`, `approve_lock` to `gates.py` (these are already mostly generic)
3. Add `maybe_gate_ready(*, project_dir, project_slug, prior_tick_id, state_dir, slack_channel, gate_key)` — generic version of ship readiness check (keyword-only params matching `maybe_ship_ready`)
4. Add `approve_gate(*, project_dir, project_slug, todo_id, state_dir, gate_key, overrides, reject_reason)` — generic version of `approve_ship`
5. Update `ship.py` to import from `gates` and delegate: `approve_ship = lambda ...: approve_gate(..., gate_key="phase_9_ship")`
6. Run existing ship-gate tests after each step — they must pass without modification
7. Add new unit tests for `maybe_gate_ready` and `approve_gate` with `gate_key="phase_2b_plan_gate"`

**Reuses:** `ship.py` `approve_lock`, `ShipSidecar`, Kanban gate registration

### Approach B: Ideal Architecture — Decision-First Pipeline

**Summary:** Build a dedicated Decision Manager module that tracks decisions from TODOS creation → Autoplan → Plan Gate → every phase → Ship Gate. Decisions are first-class entities with versioning, not just sidecars.

**Effort:** L | **Risk:** Med

**Pros:**
- Decisions are queryable, traceable, reusable across TODOs
- Enables "decision replay" — apply same decisions to similar TODOs
- Richer Slack alerts: "3 decisions carried forward, 2 overridden"

**Cons:**
- Significantly more code than Approach A
- Over-engineers for current need (one gate, one ledger)
- Migration path from current state unclear

### Approach C: Lateral/Creative — Embedded Autoplan Review

**Summary:** Don't add a separate Plan Gate phase. Make Autoplan produce a self-contained "Plan Review" artifact that the runner validates before proceeding. Runner refuses to start phase 3 unless plan passes a checksum/policy check.

**Effort:** S | **Risk:** High

**Pros:**
- No new phase, no gate machinery generalization
- Faster to implement (just validation + CLI)

**Cons:**
- Loses async Kanban approval UX (operator must be at CLI)
- No decision sheet persistence across phases
- Ship-gate pattern not reused — two different gate mechanisms
- Harder to audit: "what was approved?" lives in CI logs, not state

## Recommended Approach

**Choose A (Minimal Viable — Generic Gate Machinery)** because it reuses the battle-tested ship-gate pattern, keeps Kanban as the async approval surface, and the implementation plan already validates this approach with concrete file mappings. Codex's 48-hour prototype aligns with this approach.

## Gate Resolution (final)

- **U1 → risk-based gating.** The gate fires only for TODOs classified high-risk: dependency-changing, high-blast-radius, or `rejection_count > 0` (previously failed). Low-risk TODOs pass straight through Autoplan → Development with no gate. **Success-criteria change:** "Plan Gate triggers on 100% of TODOs" is REPLACED by "Plan Gate triggers on TODOs classified high-risk; low-risk TODOs bypass with an outcome-logged skip." A risk classifier is now in scope (see task T10). This directly answers the CEO cross-phase theme that gate-100% undermines autonomy.
- **T1 → dispatcher pre-check.** Gate logic lives in `cli._tick_project()` as a pre-runner check (mirroring `maybe_ship_ready`), NOT inside `PipelineRunner.run()`. No mid-run `return False`, no stranded tick-lock. The runner stays unaware of gates.
- **T2 → add timeout + escalation.** Configurable gate timeout in `.hermes/config.toml`; on expiry, Slack escalation (not silent stall).
- **T3 → defer ledger.** Ship the decision-sheet capture; ledger + drift detection is a follow-up TODO.
- **T4 → keep approve_ship ship-specific.** Extract only narrow lock/task-lookup primitives into `gates.py`; do not build a generic `approve_gate` transaction. Ship gate and its tests are untouched behaviorally.
