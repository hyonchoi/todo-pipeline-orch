## Problem Statement

The `hermes_pipeline` Python package has its own execution loop (`runner.py`), state machine (`state.py`), subprocess spawning (`_run_claude_subprocess()`), and direct Anthropic SDK calls (`_anthropic_call()`). It mirrors what Hermes Kanban already does well — task tracking, dependencies, execution — in a thin and error-prone way.

TODO-6 (route LLM queries through hermes) is the entry point, but the real opportunity is bigger: make the pipeline **Hermes-native** so Hermes Kanban becomes the sole execution engine. The Python package shrinks from an orchestrator to a thin bridge that reads `TODOS.md` and emits Kanban tasks.

## What Makes This Cool

The pipeline stops being a bespoke orchestrator and becomes a **declaration of intent**: read a TODO file, decompose into Kanban tasks with dependencies, let Hermes dispatch execute each phase. No more custom state machines, no more API key plumbing, no more Python subprocess management for phase execution. The work flows through the same system you'd use to manage any AI-assisted project.

## Constraints

- The current `hermes_pipeline` serves running pipelines — changes must not break in-flight TODOs.
- Hermes Kanban dispatch runs Hermes workers via `hermes chat` — the prompt structure must work in that model.
- TODO-5 (model fallback ladder) should collapse into `hermes fallback` rather than custom logic.
- A **running Hermes gateway** is required for auto-dispatch (dispatcher runs inside the gateway process, 60s tick).

## Premises

1. **Hermes Kanban is the right execution engine.** The current custom runner replicates what Hermes Kanban does natively (task state, dependencies, dispatch, error handling). Replacing it reduces code, not capability.

2. **Each pipeline phase maps to a Hermes Kanban task with dependencies.** Six Kanban phases per TODO: Plan → Plan Convert → Develop → Code Review → Document → Ship. Dependencies between phases are Kanban parent→child links, not custom code.

3. **Phases run via Hermes workers, not Python subprocesses.** Each phase task carries a body (prompt text) that the Hermes dispatcher assigns to a worker. The worker reads the body via `kanban_show()`, does the work, and calls `kanban_complete()`. The Python package doesn't spawn processes for phases — it declares tasks and watches for completion.

4. **`hermes chat -q` replaces direct Anthropic SDK calls.** The decision agent (TODO picker) routes through `hermes chat -q` instead of `anthropic.Anthropic().messages.create()`. This centralizes model policy (pinning, fallback, auth) under Hermes. Note: this is still a subprocess call from Python, but model policy shifts from Python to Hermes.

5. **The Python package becomes a Kanban task factory.** After TODO-6, the package reads `TODOS.md`, creates Kanban tasks, and monitors Kanban state. Custom `runner.py`, `state.py`, and `phases.py` subprocess logic is deleted or deprecated.

### Implications

- **Human review gate:** Use `blocked` → `promote` (P5 pattern), not `ready_for_review` → `running`. The `ready_for_review` was from the old custom `PhaseStatus` literal, not Hermes.
- **Worktree sharing:** Each child phase needs `--workspace worktree --branch feat/TODO-N` explicitly. No inheritance.
- **Circuit breaker:** Hermes has `--max-retries` and `--failure-limit` built in. The current `circuit.py` can be replaced.
- **Tick lock:** Hermes dispatcher uses claim locks with TTL. No custom tick lock needed.
- **Gstack skill resolution:** ❓ UNRESOLVED. gstack skills are Claude Code skills (`~/.claude/skills/`), not Hermes skills. The `--skill` flag pins Hermes skills. Hermes workers read the task body via `kanban_show()` and follow instructions — the body can say "run claude autoplan" and the worker would need to invoke `claude` as a subprocess. This brings back the subprocess problem for gstack skill invocation.

## Approaches Considered

### Approach A: Thin Kanban Adapter (Minimal)

Replace the custom runner with `hermes kanban create --parent` + `hermes kanban dispatch`. Each TODO becomes a Kanban parent; phases are child tasks with dependencies. Keep the Python package minimal — just read TODOS.md and emit Kanban tasks. The decision agent uses `hermes chat -q` instead of Anthropic SDK.

**Effort:** M | **Risk:** Low
**Pros:** Smallest diff, preserves existing test structure, minimal disruption.
**Cons:** Doesn't fully leverage Hermes skills. Phases still rely on Python-side prompt rendering. Partial migration — the Python package is still orchestrating, just through Kanban.

### Approach B: Full Kanban-Native Phases (Chosen)

Each pipeline phase becomes a Hermes Kanban task with a body (prompt text). Hermes dispatch executes the task — no Python subprocess spawning for phase execution. The Python package becomes a pure Kanban task factory: it creates tasks and watches for completion, but Hermes owns the execution lifecycle.

Six Kanban phases per TODO:
1. **Plan** — Body: "Run gstack autoplan skill"
2. **Plan Convert** — Body: "Run gstack writing-plan"
3. **Develop** — Body: "Run gstack subagent-driven-development"
4. **Code Review** — Body: "Run gstack /review + /qa + /cso with autofix"
5. **Document** — Body: "Run gstack document-release + document-generate"
6. **Ship** — Body: "Run gstack ship skill"

Hermes dispatch handles auth, model selection, tool access, and error handling. The Python package doesn't know about `anthropic`, `claude`, or API keys.

**Effort:** L | **Risk:** Medium (migration is a substantial rewrite; testing shape changes; gstack skill resolution unresolved)
**Pros:** Cleanest architecture. Hermes owns the full lifecycle (auth, model, tools, error handling, retry). Python package drops most of its current code. Enables TODO-7 (review phase) and TODO-8 (Kanban-gated ship) as natural extensions.
**Cons:** Migration is a bigger lift — the Python subprocess spawning is a substantial rewrite. Testing changes shape (test Kanban task creation and Hermes dispatch output, not Python function mocks). **Gstack skill resolution is unresolved** — if Hermes workers can't invoke gstack skills, the body approach needs the worker to run `claude` as a subprocess, defeating the purpose.

### Approach C: Hybrid — Hermes Dispatch + Python Bridge

Hermes Kanban tasks with dependencies, but the Python runner remains as the executor (listening for task completion events). Lower effort since we don't rewrite phases as skills yet.

**Effort:** S | **Risk:** Low
**Pros:** Fastest to implement, leverages existing Python test suite.
**Cons:** Half-measure — the Python package still owns execution, just reads Kanban for state. Doesn't solve the core problem (Python orchestrating instead of Hermes).

**Recommendation: Approach B** because it delivers the most architecturally clean state — the pipeline becomes a declaration of intent, not an execution engine. The migration is phased (see below), starting with Approach A steps and evolving into B. The **gstack skill resolution** question is the biggest unknown — if Hermes can't invoke gstack skills, we fall back to Approach C for phase execution while still using Hermes Kanban for task tracking and dependencies.

## Recommended Approach

### Target Architecture

```
TODOS.md ──► TODO Reader ──► Kanban Task Factory ──► Hermes Kanban DB
                                                       │
                                                Hermes Dispatch (gateway)
                                                (promotes todo→ready,
                                                 spawns workers per task)
                                                       │
                                                Kanban State
                                                       │
                                                 Watcher
                                                 (monitors state,
                                                  writes back to
                                                  TODOS.md)
```

The Python package creates 7 tasks per TODO (1 parent + 6 phases) via `hermes kanban create`. The Hermes gateway dispatcher handles dependency resolution (parent done → child ready), worker spawning, and completion. The watcher reads Kanban state via `hermes kanban list --status` and writes back to TODOS.md.

### Six Kanban Phases per TODO

Each TODO creates a parent task with 6 child tasks linked via `--parent`.
Each child depends on the previous child (chain-linked, not fanned out):

```bash
# Parent TODO task (board-level — no --parent needed)
hermes kanban create "TODO-N: <title>" \
  --idempotency-key todo-N

# Child phases — chain-linked: Phase N depends on Phase N-1
# All phases share the same worktree/branch explicitly (no inheritance)
BRANCH="feat/TODO-N"

hermes kanban create "Phase 1: Plan" \
  --parent <todo_task_id> \
  --workspace worktree \
  --branch "$BRANCH" \
  --body "<phase-1-prompt>" \
  --idempotency-key todo-N-phase-plan

hermes kanban create "Phase 2: Plan Convert" \
  --parent <phase1_task_id> \
  --workspace worktree \
  --branch "$BRANCH" \
  --body "<phase-2-prompt>" \
  --idempotency-key todo-N-phase-plan-convert

hermes kanban create "Phase 3: Develop" \
  --parent <phase2_task_id> \
  --workspace worktree \
  --branch "$BRANCH" \
  --body "<phase-3-prompt>" \
  --idempotency-key todo-N-phase-develop

hermes kanban create "Phase 4: Code Review" \
  --parent <phase3_task_id> \
  --workspace worktree \
  --branch "$BRANCH" \
  --body "<phase-4-prompt>" \
  --idempotency-key todo-N-phase-review

hermes kanban create "Phase 5: Document" \
  --parent <phase4_task_id> \
  --workspace worktree \
  --branch "$BRANCH" \
  --body "<phase-5-prompt>" \
  --idempotency-key todo-N-phase-document

hermes kanban create "Phase 6: Ship" \
  --parent <phase5_task_id> \
  --workspace worktree \
  --branch "$BRANCH" \
  --initial-status blocked \
  --body "<phase-6-prompt>" \
  --idempotency-key todo-N-phase-ship
```

Phase 6 (Ship) is created with `--initial-status blocked` — it will be promoted by the operator after review.

The `--parent` chain creates a linear dependency: Phase N waits for Phase N-1 to complete. Hermes dispatch automatically runs tasks whose parent tasks are done — it promotes `todo → ready` when all parents are `done`.

The parent TODO task is a container — after creation, the factory marks it `done` immediately. This unblocks Phase 1 (its only child with `--parent <todo_task_id>`) for the dispatcher. The parent TODO's `done` status is metadata only; it does not mean the overall TODO is complete — the watcher tracks Phase 6 for that.

**Linearity and failure:** If Phase 3 fails, unreached child tasks (4, 5, 6) remain in `todo` status. On restart, the Kanban factory detects existing tasks via `--idempotency-key` and skips recreation. The watcher sees Phase 3 is `failed` and can either retry (if transient) or block (if terminal). Hermes dispatch has `--failure-limit` (default 2) for auto-blocking on consecutive non-success attempts.

### Worktree Sharing Across Phases

Each child phase needs `--workspace worktree --branch feat/TODO-N` explicitly — no inheritance across tasks. The first phase (Plan) creates the branch. Subsequent phases check out the same branch via `--branch`. Since Hermes dispatches workers sequentially (chain-linked), by the time Phase 2 runs, the branch from Phase 1 already exists.

If the branch doesn't exist yet (e.g., Phase 1 failed to create it), Phase 2 should create it. The body prompt should include: "If branch feat/TODO-N doesn't exist, create it from main."

### Human Review Gate (Phase 5 → 6)

After Phase 5 (Document) completes, Phase 6 (Ship) is gated by human review.
The gate uses the P5 Human-in-the-Loop pattern: `blocked` → `promote`.

1. Phase 5 completes. Hermes dispatch marks it `done`. Because Phase 6 was created with `blocked` initial status, parent completion does **not** auto-promote it — Phase 6 remains `blocked`.
2. Operator reviews the branch (a `git diff main...feat/TODO-N --stat` is enough) and runs `hermes kanban promote <phase6_task_id> "Operator approved for ship"`.
3. Hermes dispatch picks up the now-`ready` Phase 6 and spawns a worker.

This is the documented P5 Human-in-the-Loop pattern from Hermes Kanban. No custom `ready_for_review` status needed.

### Abort and Rollback

When a TODO needs to be aborted mid-pipeline:

1. **Operator abort:** `hermes kanban block <todo_task_id>` blocks the parent. Children in `todo` stay blocked (their parent isn't `done`). Children in `running` are reclaimed on the next dispatch tick.
2. **Automatic abort on failure:** If a phase hits `--max-retries N` failures, Hermes dispatch marks it `failed` and blocks downstream tasks via `--failure-limit`.
3. **Rollback:** No automatic rollback. If Phase 4 (Code Review) finds issues that warrant abandoning the work, the operator blocks the TODO and creates a new TODO with corrective instructions. The branch is preserved for reference (not force-deleted).

### TODOS.md Write-Back

When a TODO completes (Phase 6 done), the watcher writes back to `TODOS.md`:
- Updates the status marker from `[ ]` to `[x]`
- Appends a completion note with the branch name and merge commit

When a TODO fails, the watcher writes `[~]` (on hold) and appends the failure reason.

### Kanban Task Prompts (Phase Bodies)

Each task carries a body that the Hermes worker reads via `kanban_show()`. The body contains instructions. The worker follows them using its available tools and skills.

**Phase 1 — Plan:**
```
Run the gstack autoplan skill.
1. Read TODOS.md and confirm the in-flight TODO is TODO-N.
2. Run CEO/Eng/UI/DX reviews.
3. Write the plan into .hermes/plans/.
4. Create a new branch feat/TODO-N from main and commit the plan.
5. Save the branch name to .hermes/pipeline_branch.txt.

```

**Phase 2 — Plan Convert:**


**Phase 3 — Develop:**
```

```

**Phase 4 — Code Review:**
```

```

**Phase 5 — Document:**
```

Project root: ~/projects/<project>
```

**Phase 6 — Ship:**
```
Run gstack ship skill. Merge to main (no PR).

Project root: ~/projects/<project>
```

### Gstack Skill Resolution — The Open Question

The bodies above say "Run the gstack autoplan skill" etc. These are Claude Code skills, not Hermes skills. How does a Hermes worker execute them?

**Option A — Hermes invokes claude as a subprocess:** The body instructs the Hermes worker to run `claude -p "Run gstack autoplan skill..."` as a subprocess. This brings back the subprocess problem the design is trying to eliminate.

**Option B — Convert gstack skills to Hermes skills:** If gstack skills can be installed as Hermes skills (via `hermes skills install`), then `--skill gstack-autoplan` pins them to the task, and the worker can call them directly. This is the cleanest path.

**Option C — Inline the prompt in the body:** Instead of referencing a skill, the body contains the full prompt text. This is what the current `configs/phases.yaml` does. The body IS the prompt.

**Recommendation:** Start with Option C (inline body). The bodies above ARE the prompts. If the body is long enough to warrant a skill, convert to Option B. Avoid Option A as it brings back subprocess management.

### Decision Agent via `hermes chat -q`

The current `_anthropic_call()` in `decision/agent.py` is replaced:

```python
# Before: direct Anthropic SDK
def _anthropic_call(*, model: str, max_tokens: int, prompt: str) -> str:
    import anthropic
    client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
    msg = client.messages.create(
        model=model, max_tokens=max_tokens, messages=[{"role": "user", "content": prompt}],
    )
    return "".join(b.text for b in msg.content if getattr(b, "type", "") == "text")

# After: route through Hermes (still a subprocess, but policy shifts)
def _hermes_call(*, prompt: str, model: str | None = None) -> str:
    cmd = ["hermes", "chat", "-q", prompt, "-Q"]
    if model:
        cmd.extend(["-m", model])
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=300)
    if result.returncode != 0:
        raise RuntimeError(f"hermes chat failed: {result.stderr[:200]}")
    return result.stdout
```

Note: this is still a subprocess call from Python. The improvement is that model policy, auth, and fallback are managed by Hermes rather than the Python package. The `anthropic` package is removed from Python dependencies.

## Reviewer Concerns

The following concerns persisted across all 4 rounds of adversarial review and were resolved by Phase 0:

- **Hermes auto-dispatch:** ✅ CONFIRMED — dispatcher promotes `todo → ready` when all parents are `done`.
- **CLI flag assumptions:** ✅ CONFIRMED — `--parent`, `--idempotency-key`, `--workspace`, `--branch`, `--skill`, `--max-retries` all exist on `hermes kanban create`.
- **Worktree inheritance:** ❌ NO INHERITANCE — each child needs explicit `--workspace worktree --branch`.
- **Human review gate:** ❌ NO `ready_for_review` — use `blocked` → `promote` (P5 pattern).

Remaining concerns:

- **Gstack skill resolution (fatal if wrong):** gstack skills are Claude Code skills (`~/.claude/skills/`), not Hermes skills. If Hermes workers can't invoke them, the body approach needs Option C (inline body) or Option A (subprocess).
- **`hermes chat -q` output compatibility:** The current `_parse()` function expects raw text from the Anthropic SDK. `hermes chat -q` may add markdown wrapping, timestamps, thinking tokens, or other formatting that breaks the parser. Needs empirical testing.
- **Testing migration cost:** Mocking 7+ different Hermes CLI commands per TODO lifecycle is harder than mocking Python function calls.
- **Gateway dependency:** The Hermes gateway must be running for auto-dispatch. This is a new operational requirement not present in the current system.
