## Problem Statement

The pipeline orchestrator runs unattended kanban phases (autoplan, implementation, code review, security review, ship) through a Claude Code agent launched by Hermes. Two gaps prevent this from working well:

1. **No declarative contract** — What tools, capabilities, and assignee the agent needs are hardcoded or inferred. When phases.yaml evolves to require a new tool, the pipeline silently degrades because nobody verified the config matches reality.

2. **No dedicated agent personality** — The agent runs with whatever profile the operator configured for interactive chat. A general-purpose chat assistant is the wrong tuning for autonomous, unattended code execution. No SOUL.md means no guardrails against interactive prompts, verbose narration, or indecisive behavior during a phase that says "apply fixes."

## What Makes This Cool

A single `pipeline-watch init <project>` command wires up the entire unattended pipeline: writes the execution contract, installs the Hermes profile with SOUL.md, and sets the assignee. The agent shows up for work like a professional — not a chatbot.

## Current State (What Exists)

The contract module ships with passing tests (30 tests, 3 files):

- `hermes_pipeline/contract.py` — `PipelineContract(schema_version, assignee, capabilities)` dataclass, loader, validator, error hierarchy
- `pipeline-watch init <project>` — Writes default contract to project state, idempotent with `--force`
- `pipeline-watch doctor <project>` — Validates contract against phases.yaml, exit codes 0/1/2
- Tick flow integration — Contract loaded at tick start, `assignee` passed to `register_todo_phases()`, capability mismatch raises and skips project

**What's missing:** Hermes profile with SOUL.md, distribution packaging, documentation updates, stale impl plan sync, TODO-15 not marked done.

## Constraints

- Python 3.12+, uv-managed package
- No new runtime dependencies
- Contract is per-project at `<project_state>/pipeline.toml`
- Profile lives in `~/.hermes/` via Hermes profile API
- Hermes profile create: `hermes profile create <name> [--description] [--clone] [--no-alias]`
- Hermes profile install: `hermes profile install <source>` (git URL or local dir with `distribution.yaml`)
- SOUL.md is the profile's personality file — loaded into agent context on every invocation
- Backward compatible — existing ticks work without a contract (fallback to `"default"` assignee)

## Premises

1. **The default profile causes unexpected behavior in unattended mode** — The operator's profile is tuned for interactive chat. Running autonomous kanban phases through it risks interactive prompts, verbose narration, and indecisive behavior. Accept with refinement: reasonable for P2, even without concrete failure data — it's a mismatch of use cases.

2. **SOUL.md is the right mechanism for agent personality** — It's the only profile-level surface for shaping agent behavior. Skills provide instructions; SOUL.md provides overarching tone and guardrails.

3. **The profile should be bundled with the repo** — Versioned with the code, installable via `pipeline-watch init` or `hermes profile install`. An operator who wants a custom SOUL.md can always `hermes profile create` and edit by hand.

4. **The agent should be a "silent professional"** — Executes phases methodically, follows skill instructions literally, surfaces errors without dwelling, decisive when prompts say "apply fixes." No chat-style narration. Wastes no tokens on conversation nobody reads.

## Approaches Considered

### Approach A: Just Design SOUL.md Content

Write the SOUL.md and stop. Manual profile creation.
- **Pros:** Fast — brainstorm the personality and write the doc
- **Cons:** Leaves manual work for every operator. No way to install it as a default.

### Approach B: Design SOUL.md + Installable Distribution (Recommended)

Create SOUL.md, package it as an installable Hermes distribution (`profiles/pipeline/` directory with `distribution.yaml`), and wire `pipeline-watch init` to offer profile installation.
- **Pros:** Covers the hard part (SOUL.md design) and makes it deployable. Bundled in repo, versioned with code. `pipeline-watch init` or `hermes profile install` gets it live.
- **Cons:** More moving parts — distribution.yaml, profile directory structure.

### Approach C: Full Automation — Profile Created by Init Command

`pipeline-watch init` calls `hermes profile create pipeline`, writes SOUL.md directly, sets description, and wires assignee automatically.
- **Pros:** One-command onboarding — no manual steps.
- **Cons:** Most effort for what is fundamentally a text file. Risks over-engineering. Couples to Hermes profile CLI flags that may change.

## Recommended Approach

**B: Design SOUL.md + Installable Distribution.**

The SOUL.md is the hard part — getting the agent personality right. Distribution packaging is mechanical. B captures both without the over-engineering risk of C.

The `profiles/pipeline/` directory in the repo contains:
- `distribution.yaml` — Hermes distribution manifest
- `SOUL.md` — Agent personality

Installation: either `hermes profile install profiles/pipeline/` manually, or `pipeline-watch init <project>` can offer to install it.

## SOUL.md Design

The pipeline agent has a specific job, reflected in the personality:

**Role:** Unattended worker driving kanban phases autonomously.

**Phases it executes:**
1. Phase 2: Autoplan (gstack autoplan skill) — 20 turns, 30 min
2. Phase 3: Writing Plan (writing-plans skill) — 15 turns, 30 min
3. Phase 4: Development (subagent-driven-development) — 60 turns, 60 min
4. Phase 5: Code Review (gstack /review skill) — autonomous mode, 30 turns, 40 min
5. Phase 6.1: CSO Security Review (gstack cso skill) — 20 turns, 30 min
6. Phase 7: Document Release — 15 turns, 30 min
7. Phase 8: Finish Branch (finishing-a-development-branch) — 15 turns, 30 min

**Key behaviors:**
- No interactive prompts. There is no human at the terminal.
- Follows skill instructions literally — the phase prompt is the spec, not a suggestion.
- Commits are its voice — clear messages, atomic changes.
- Surfaces errors but doesn't dwell. If a phase prompt says "apply fixes," it applies fixes. No "should I?"
- Narrates only what's necessary for debugging — a phase stall should be diagnosable from the output, but a successful phase should be terse.
- Decisive on judgment calls: when reviewing code, it decides; when writing, it writes; when shipping, it ships (or halts at a gate, as instructed).
