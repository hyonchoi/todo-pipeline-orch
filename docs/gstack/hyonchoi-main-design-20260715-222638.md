## Problem Statement
The mock integration test harness (`hermes_pipeline/harness.py`, shipped by TODO-19)
hardcodes `NullKanbanAdapter()` at `harness.py:357` when constructing the
`PipelineRunner`. The harness already drives real `hermes`/`claude` subprocesses
through phase execution end-to-end, but kanban sync (task creation, phase comments,
complete/archive) is always a silent no-op. That's the last mocked system in an
otherwise real end-to-end harness.

## What Makes This Cool
It closes the loop. TODO-19 built a harness that reproduces real cross-system
pipeline behavior deterministically — this makes kanban the last piece that's
actually real too, so a harness run can now surface genuine kanban-sync bugs
(task creation failures, comment formatting, outbox retry behavior) the same way
it already surfaces phase-execution bugs.

## Constraints
- Must not change default harness behavior — CI and hermetic runs must stay
  unaffected (`--kanban null` is the default, matching today's hardcoded behavior).
- Must not require network/tenant access for the default path.
- Effort should stay within the TODO's original S-sizing.

## Approaches Considered

### Approach A: Minimal Flag (harness-only slug suffix) — Chosen
Add `--kanban {null,hermes}` to the `test` subcommand. When `hermes` is selected,
the harness generates a unique project slug suffix internally
(`f"{fixture['project_slug']}-{tick_id}"`, reusing the `tick_id` already generated
at `harness.py:370`) and constructs `HermesKanbanAdapter(outbox, active_tasks)`
with real `KanbanOutbox`/`ActiveTasksStore` under `state_dir`, instead of
`NullKanbanAdapter()`.
- Effort: S
- Risk: Low
- Pros: No new CLI knob beyond the flag itself; reuses an ID scheme that already
  exists and is already unique per run; smallest possible diff.
- Cons: Cards on the real board show suffixed slugs like `mock-project-a1b2c3`,
  slightly less readable when eyeballing the board by name alone.
- Reuses: `tick_id` generator (`logging_setup.new_tick_id`), `HermesKanbanAdapter`,
  `KanbanOutbox`, `ActiveTasksStore` — all pre-existing, no new abstractions.

### Approach B: Explicit `--kanban-project` override
Same wiring as A, plus an optional `--kanban-project NAME` flag letting the user
pin a stable, readable project name across multiple runs of the same debug
session.
- Effort: S/M
- Risk: Low
- Pros: Lets the user deliberately reuse a name across runs when chasing one bug
  (matches the "orphaned cards as debug trail" workflow); more control.
- Cons: A second CLI flag to document and test; user must remember to set it when
  continuity across runs matters.
- Reuses: same underlying wiring as A.

### Approach C: Config-driven kanban mode
Move kanban-adapter selection into `.hermes/config.toml` (`[test].kanban_adapter`)
instead of a CLI flag, read via the existing `Config` object already threaded
into `run_harness(config=...)`.
- Effort: M
- Risk: Med
- Pros: Consistent with how other pipeline behavior is configured elsewhere in
  the repo.
- Cons: Config edits are heavier than a flag for what's fundamentally a one-off
  debug action; doesn't match the TODO's explicit ask for a CLI flag; harder to
  toggle per-invocation.
- Reuses: `Config` plumbing.

## Recommended Approach
**Approach A.** It satisfies TODO-20 exactly as scoped, closes the collision gap
found in cross-model review for free (reusing `tick_id`, not inventing a new ID
scheme), and requires no new CLI surface beyond the single flag the TODO already
names.

## Open Questions

### D1 — Stop conflating tenant and run identity

**Decision:** Keep one fixed kanban tenant (`fixture["project_slug"]`, unsuffixed —
`"mock-project"` today) as the only value ever passed as `--tenant`. The per-run `tick_id`
never touches `--tenant`; it goes into the card **body** as metadata instead.

This directly closes Eng Finding 1 (`kanban.py:339` — `project` flows straight into
`--tenant`; suffixing it was creating a new tenant per run, not a labeled card in one
tenant).

**Consequence for `ActiveTasksStore`:** `ActiveTasksStore.set(project, task_id)` /
`.get(project)` (`kanban.py:107-127`) is keyed by `project`. Since `project` is now fixed
across runs (not suffixed), two overlapping `--kanban hermes` invocations against the same
`state_dir` would still collide on the active-task lookup key — but each harness invocation
gets its own fresh temp `state_dir` (`harness.py:338`), so `ActiveTasksStore` itself is
never actually shared across runs. Collision was never really an `ActiveTasksStore` problem;
it was a `--tenant` problem. No `ActiveTasksStore` changes needed.

**Card body now carries:** `tick_id`, fixture name, phase, TODO ID — richer than the current
`Phase: {phase}\nTODO ID: {todo_id}` (`kanban.py:335`), closing Eng Finding 6. Reuses the
JSON-header pattern already present in `kanban_tasks.py:123` rather than inventing a new
format.

### D2 — Reconcile "orphaned cards" premise with actual archive-on-failure behavior

**Decision:** Accept archive-on-failure as-is (Eng Finding 5's actual runner behavior is
correct and stays unchanged). Debug trail = archived cards + any still-live cards together,
not "everything stays live forever." This directly walks back original Premise 2 ("no
cleanup logic should be added") — cleanup logic already exists (`clear_active_task`
archives on non-merged outcomes) and this revision keeps it, rather than suppressing it.

**Terminal-state table** (closes Eng Finding 5's request):

| Terminal state | Runner path | Board state |
|---|---|---|
| Success (phase merged) | `clear_active_task(outcome="merged")` | Card **completed** |
| Phase failure, `continue_on_failure=True` | `clear_active_task(outcome="failed")` | Card **archived** (inspectable, not deleted) |
| Convergence-halt | same as phase failure | Card **archived** |
| Overall timeout | bypasses `clear_active_task` (runner never reaches it) | Card **live** — genuinely orphaned |
| Process crash | bypasses `clear_active_task` | Card **live** — genuinely orphaned |

Only timeout/crash paths produce a truly live orphaned card; failure/convergence-halt
produce an archived-but-inspectable one. Both are legitimate debug signal — a live card
means "the run never got to clean up," an archived card means "it failed cleanly and here's
why." The doc's problem statement and success criteria below are updated to describe this
distinction rather than treating "orphaned" as the only outcome.

### D3 — Approach B (`--kanban-project`/`--kanban-tenant` override)

**Decision:** Not needed. D1's fixed-tenant fix already gives the single stable location
Approach B was trying to provide — every `--kanban hermes` run lands in the same tenant with
a `tick_id` label in the body, which is what "grouped breadcrumb trail" meant. Approach B is
formally dropped, not deferred; nothing here depends on it being revisited later unless a
future need for multiple isolated tenants (e.g. two people debugging in parallel against
different boards) actually materializes.
