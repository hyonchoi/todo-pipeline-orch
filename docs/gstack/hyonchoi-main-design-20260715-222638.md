## Problem Statement
The mock integration test harness (`hermes_pipeline/harness.py`, shipped by TODO-19)
hardcodes `NullKanbanAdapter()` at `harness.py:357` when constructing the
`PipelineRunner`. The harness already drives real `hermes`/`claude` subprocesses
through phase execution end-to-end, but kanban sync (task creation, phase comments,
complete/archive) is always a silent no-op. That's the last mocked system in an
otherwise real end-to-end harness.

## What Makes This Cool
It closes the loop. TODO-19 built a harness that reproduces real cross-system
pipeline behavior deterministically — this makes kanban the last piece that's
actually real too, so a harness run can now surface genuine kanban-sync bugs
(task creation failures, comment formatting, outbox retry behavior) the same way
it already surfaces phase-execution bugs.

## Constraints
- Must not change default harness behavior — CI and hermetic runs must stay
  unaffected (`--kanban null` is the default, matching today's hardcoded behavior).
- Must not require network/tenant access for the default path.
- Effort should stay within the TODO's original S-sizing.

## Cross-Model Perspective
Independent cold read (Claude subagent, dispatched after Codex timed out at 5min):

- **Steelman**: The goal is unlocking end-to-end integration testing where real
  pipeline phase execution AND kanban sync side effects happen in a single test
  run against a dedicated tenant. Toggling the adapter via `--kanban` gives fast/
  hermetic runs by default, with an opt-in path to real kanban validation when

- **Key insight** (quoting the session): *"orphaned kanban cards left behind on

  pipeline failure modes." That reframing is what surfaced the collision risk
  below, since breadcrumbs from different runs need to stay distinguishable.
- **Challenged premise**: "Scope stays Effort S with no other changes" — the
  subagent flagged that `project_slug` isn't unique per run, so repeated
  `--kanban hermes` invocations against the same tenant would collide in
  `ActiveTasksStore`. Verified directly against the code (`harness.py:44`,
  hardcoded to `"mock-project"`) — the concern was correct. Premise 4 revised
  above to close the gap using the harness's existing `tick_id`.
- **Prototype suggestion**: construct the real adapter only when `--kanban hermes`
  is passed, wire it to `KanbanOutbox`/`ActiveTasksStore` paths under the fixture's
  `state_dir`, and skip all cleanup/retry-tuning/tenant-preflight work as
  out-of-scope for this TODO.

## Approaches Considered

### Approach A: Minimal Flag (harness-only slug suffix) — Chosen
Add `--kanban {null,hermes}` to the `test` subcommand. When `hermes` is selected,
the harness generates a unique project slug suffix internally
(`f"{fixture['project_slug']}-{tick_id}"`, reusing the `tick_id` already generated
at `harness.py:370`) and constructs `HermesKanbanAdapter(outbox, active_tasks)`
with real `KanbanOutbox`/`ActiveTasksStore` under `state_dir`, instead of
`NullKanbanAdapter()`.
- Effort: S
- Risk: Low
- Pros: No new CLI knob beyond the flag itself; reuses an ID scheme that already
  exists and is already unique per run; smallest possible diff.
- Cons: Cards on the real board show suffixed slugs like `mock-project-a1b2c3`,
  slightly less readable when eyeballing the board by name alone.
- Reuses: `tick_id` generator (`logging_setup.new_tick_id`), `HermesKanbanAdapter`,
  `KanbanOutbox`, `ActiveTasksStore` — all pre-existing, no new abstractions.

### Approach B: Explicit `--kanban-project` override
Same wiring as A, plus an optional `--kanban-project NAME` flag letting the user
pin a stable, readable project name across multiple runs of the same debug
session.
- Effort: S/M
- Risk: Low
- Pros: Lets the user deliberately reuse a name across runs when chasing one bug
  (matches the "orphaned cards as debug trail" workflow); more control.
- Cons: A second CLI flag to document and test; user must remember to set it when
  continuity across runs matters.
- Reuses: same underlying wiring as A.

### Approach C: Config-driven kanban mode
Move kanban-adapter selection into `.hermes/config.toml` (`[test].kanban_adapter`)
instead of a CLI flag, read via the existing `Config` object already threaded
into `run_harness(config=...)`.
- Effort: M
- Risk: Med
- Pros: Consistent with how other pipeline behavior is configured elsewhere in
  the repo.
- Cons: Config edits are heavier than a flag for what's fundamentally a one-off
  debug action; doesn't match the TODO's explicit ask for a CLI flag; harder to
  toggle per-invocation.
- Reuses: `Config` plumbing.

## Recommended Approach
**Approach A.** It satisfies TODO-20 exactly as scoped, closes the collision gap
found in cross-model review for free (reusing `tick_id`, not inventing a new ID
scheme), and requires no new CLI surface beyond the single flag the TODO already
names.

## Open Questions
- None blocking. If a future session wants stable cross-run naming for a specific
  bug hunt, revisit Approach B as a follow-on (not in scope here).

## Success Criteria
- `hermes-pipeline test --fixture <name> --kanban hermes` constructs a real
  `HermesKanbanAdapter` wired to `KanbanOutbox`/`ActiveTasksStore` under the
  fixture's temp `state_dir`, with a `tick_id`-suffixed project slug.
- `hermes-pipeline test --fixture <name>` (no `--kanban` flag, or `--kanban null`)
  behaves identically to today — `NullKanbanAdapter()`, no network calls.
- Running `--kanban hermes` twice in a row against the same tenant produces two
  distinct cards/task entries, not a collision.
- `docs/howto-mock-integration-test-harness.md` gets a new "Run with real kanban
  adapter" step documenting the flag and the tenant precondition.

## Distribution Plan
No new distribution surface — this ships as part of the existing `hermes-pipeline`
CLI package via the normal version-bump/release flow (VERSION + pyproject.toml +
uv.lock + CHANGELOG, per this repo's CLAUDE.md).

## Next Steps
1. Add `--kanban {null,hermes}` argument to `test_parser` in `cli.py` (near
   line 536-565), default `"null"`.
2. Thread `kanban_mode: str` through `_cmd_test` → `run_harness()`.
3. In `run_harness()` (replacing the hardcoded line at `harness.py:357`): branch
   on `kanban_mode` — `"null"` keeps `NullKanbanAdapter()`; `"hermes"` constructs
   `KanbanOutbox(state_dir / "kanban_outbox.jsonl")` and
   `ActiveTasksStore(state_dir / "active_tasks.json")`, builds the suffixed
   project slug via `tick_id`, and constructs
   `HermesKanbanAdapter(outbox, active_tasks)`. Both files live under `state_dir`
   (created once per `run_harness()` call, before the phase loop) and are subject
   to the same `--keep`/temp-dir cleanup as the rest of `state_dir` — no separate
   lifecycle.

   Note on `--loop`: `--loop` is not an in-process repeat — each CLI invocation of
   `hermes-pipeline test` calls `run_harness()` exactly once, generating one fresh
   `tempfile.mkdtemp()` (`harness.py:338`) and one fresh `tick_id` (`harness.py:370`)
   per invocation, unconditionally. `--loop`'s only effect (`harness.py:434-446`) is
   to diff this invocation's `report.json` against the *previous separate
   invocation's* persisted `{fixture_name}-report.N.json` file on disk. So there is
   no ambiguity to resolve: every `--kanban hermes` invocation — looped or not —
   gets its own unique `tick_id`-suffixed project slug and its own fresh
   outbox/store files, by construction. Re-running with `--loop` will still
   produce distinct kanban cards per invocation (accumulating on the board across
   loop runs), matching the intended debug-trail use case.
4. Add tests:
   - `--kanban null` (default) produces zero subprocess calls to `hermes kanban`.
   - `--kanban hermes` constructs the real adapter with a `tick_id`-suffixed slug.
   - Two separate `--kanban hermes` invocations (simulating repeated/looped runs)
     produce two distinct suffixed slugs and two distinct `active_tasks.json`
     files under their own temp `state_dir`s — verifying no collision, without
     requiring a live tenant (assert on the constructed slug/paths, not on real
     `hermes kanban` network calls).
5. Document the new flag and tenant precondition in
   `docs/howto-mock-integration-test-harness.md`.
