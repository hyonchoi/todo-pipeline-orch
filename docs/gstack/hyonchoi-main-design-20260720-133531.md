# Design: Extract Plan-Gate Status Read into `gate_state.py`

Generated by /improve-codebase-architecture + /grilling on 2026-07-20
Branch: main
Repo: todo-pipeline-orchestrator
Status: DRAFT — pending review
Supersedes: (none — first design doc for this deepening)

## Problem Statement

The plan-gate's read-side status logic — "is this gate blocked, ready, running,
failed, or unknown, given kanban status and the rejection sidecar?" — is
implemented twice. `gates.check_gate_status()` is the canonical version:
rejection sidecar takes priority, else read kanban status, mapping kanban's
`"done"` to `GateStatus.RUNNING`. `kanban_tasks.all_phases_complete()`
re-implements the sidecar-priority rule inline for the single case of
`phase_2b_plan_gate`, using a local `from .gates import read_rejection_sidecar`
to dodge a circular import (`gates.py` already imports from `kanban_tasks.py`
at module level for `get_todo_kanban_tasks`/`get_todo_kanban_status`).

Two implementations of "does a rejection sidecar mean this gate is resolved"
means a future change to that rule (e.g. adding a rejection-count cap) has to
be made in two places, and nothing enforces that it is. This module was
flagged during an `/improve-codebase-architecture` pass as a **shallow seam**:
`kanban_tasks.py`'s special case doesn't add new behavior, it duplicates
`gates.py`'s.

## What Makes This Cool

It's a small, surgical move, not a rewrite: one function moves to a new file,
one call site swaps an inline duplication for a function call, and the
circular-import workaround moves with it rather than multiplying. The
deletion test confirms it's worth doing — deleting `all_phases_complete`'s
inline sidecar check doesn't remove behavior, it just makes the function fall
through to treating a rejected plan-gate as still-blocked, which is the exact
bug this checklist-adjacent duplication risks introducing silently the next
time `check_gate_status`'s rule changes and nobody remembers to update the
copy.

## Constraints

- **Scope: plan-gate only.** The ship-gate (`phase_9_ship`) has no dedicated
  status-read function today — it piggybacks on `all_phases_complete`'s
  generic blocked-phase handling — and is out of scope. Generalizing to "any
  blockable kanban phase" is also out of scope; nothing today needs a third
  gate type, and speculative generality isn't a deepening, it's a shallow
  abstraction over one call site.
- **Read-only module.** `gate_state.py` exposes a status *read* only. Writes
  to the rejection sidecar (`gates.write_rejection_sidecar`) and kanban
  status mutation (via the `hermes kanban` subprocess, invoked from
  `approve_plan.py`) stay exactly where they are today. These are a real
  **adapter** boundary — `hermes kanban` is a separate process this codebase
  doesn't own — not an accidental scatter to be collapsed.
- **`maybe_plan_gate_ready` stays in `gates.py`, unmoved.** It has a
  `slack.notify` side effect and reads three additional sources (decision
  sheet, non-gate phase completion) that `check_gate_status` doesn't touch.
  Merging it into the new read-only module would give that module two
  different jobs (status query vs. readiness-and-notify trigger) behind one
  seam. It stays a sibling, not a resident.
- **`kanban_tasks.all_phases_complete`'s special case is replaced, not kept
  alongside.** Its inline `phase_2b_plan_gate` branch (reading
  `read_rejection_sidecar` directly, with the circular-import comment) is
  deleted and replaced with a call to the new module's status function. This
  is the one behavior change in this design — everywhere else is a pure move.
- **No new class.** `GateStatus` (the enum) and the status function have no
  instance state — every parameter (`state_dir`, `project_slug`, `tick_id`,
  `gate_key`) is passed per call today, at three call sites
  (`cli.py`, `phases.py`, and now `kanban_tasks.py`). A `GateStateMachine`
  class binding these at construction was considered and rejected: it adds
  ceremony without removing any parameter repetition that isn't already
  minimal, and it invites a false impression that the module holds mutable
  state, when it strictly doesn't.
- **Circular import: same pattern as today, moved.** `kanban_tasks.py`'s
  `all_phases_complete` gets a local `from .gate_state import gate_status`
  inside the function body — matching the existing local-import workaround,
  just pointed at the new module instead of `gates.py`. `gate_state.py`
  itself imports `kanban_tasks.get_todo_kanban_tasks` at module level (safe,
  since `kanban_tasks.py` never imports `gate_state` at module level).
- **Tests update in place.** `tests/test_runner_gate.py` is today's only test
  file exercising `check_gate_status`/`GateStatus`; its imports and mocks move
  to `gate_state.gate_status`. No new dedicated test file — the module is one
  function plus one enum, small enough to test alongside its callers. Add one
  new test asserting `all_phases_complete` correctly resolves a rejected
  plan-gate via the new function (this path is exercised today only
  indirectly, through the inline special case).

## What Changes

**New file `hermes_pipeline/gate_state.py`:**
- `GateStatus` enum (moved verbatim from `gates.py`): `BLOCKED`, `READY`,
  `RUNNING`, `FAILED`, `UNKNOWN`.
- `gate_status(*, state_dir, project_slug, gate_key=PLAN_GATE_PHASE_KEY,
  tick_id) -> GateStatus` (moved verbatim from `gates.check_gate_status`,
  renamed to drop the redundant "check_" prefix now that it lives in a module
  named for exactly this concern).

**`hermes_pipeline/gates.py`:**
- Removes `GateStatus` and `check_gate_status`.
- Keeps `write_rejection_sidecar`, `read_rejection_sidecar`, `is_high_risk`,
  `maybe_plan_gate_ready`, decision-sheet read/write functions, and
  `PLAN_GATE_PHASE_KEY`. `gate_state.py` imports `PLAN_GATE_PHASE_KEY` from
  `gates.py` as the default `gate_key`.
- `maybe_plan_gate_ready` is unchanged in body, but now calls
  `gate_state.gate_status` in place of `check_gate_status` where it currently
  reads gate state (if it does today — verify at implementation time; if it
  reads `get_todo_kanban_status` directly instead, no change needed there).

**`hermes_pipeline/kanban_tasks.py`:**
- `all_phases_complete`'s `phase_2b_plan_gate` special case (the block
  starting `if key == "phase_2b_plan_gate":` with the inline
  `from .gates import read_rejection_sidecar as _rej_read` import) is
  replaced with a local import of `gate_status` and a check against
  `GateStatus.FAILED` in place of the direct sidecar read.

**Call sites updated:**
- `hermes_pipeline/cli.py` — import of `gates.check_gate_status` (if present)
  becomes `gate_state.gate_status`.
- `hermes_pipeline/phases.py:275-277` — `from .gates import GateStatus,
  check_gate_status` becomes `from .gate_state import GateStatus, gate_status`.

**Tests:**
- `tests/test_runner_gate.py` — update imports/mocks from
  `gates.check_gate_status`/`gates.GateStatus` to
  `gate_state.gate_status`/`gate_state.GateStatus`.
- Add one test covering `all_phases_complete`'s rejected-plan-gate path via
  the new shared function, replacing implicit-only coverage of the deleted
  inline branch.

## Non-Goals

- No change to the ship-gate or any other kanban phase's blocking behavior.
- No change to where rejection-sidecar or kanban-status *writes* happen.
- No new abstraction for "any blockable gate" — this is scoped to the one
  gate that has this duplication today.
- No change to `maybe_plan_gate_ready`'s Slack-notify responsibility or
  timing.

## Open Question for Implementation

Confirm at implementation time whether `maybe_plan_gate_ready` currently
calls `check_gate_status` anywhere in its body, or only reads
`get_todo_kanban_status` directly (the `find_symbol` read during design
showed it using `get_todo_kanban_status` + a direct `BLOCKED`/
`COMPLETION_STATUSES` comparison, not `check_gate_status`) — if so, no edit
is needed there beyond leaving it in `gates.py` unmoved.
