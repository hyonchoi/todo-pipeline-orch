# Design: TODO Pipeline Orchestrator — Selection, Versioning, Kanban & Merge-Gating

Generated by /office-hours on 2026-06-10

## Problem Statement

`docs/pipeline-modularization-plan.md` already specs out modularizing `pipeline_watcher.py`
into a `hermes_pipeline` package: a cron job (`pipeline-watch --auto`, every 5 min) scans
project directories for `TODOS.md`, and on change runs a Phase-based pipeline (Autoplan →
Writing Plan → Development → CSO Security Review → Document Release → Finish Branch) via
`claude -p` against the chosen TODO.

That plan left five things unspecified, which this session refines:

1. **Selection policy** — which TODO to work on next, given priority (P1-P4), effort
   (S/M/L/XL), inter-TODO dependencies, and "does this set up the next iteration."
2. **Version policy** — what `{version}` means in `feat/{version}-{slug}` branches, and
   when a "final version" gets decided and written to `VERSION`/`CHANGELOG.md`.
3. **Kanban integration** — surfacing the **currently-running pipeline task** for each
   project on the kanban plugin in the user's existing **hermes agent framework** (runs
   on a remote server; interface not inspectable from this machine).
4. **Concurrency** — exactly one TODO in flight per project; multiple projects run in
   parallel, each with its own lock.
5. **Merge gating** — the pipeline never auto-merges. It stops after Phase 8 with a PR
   ready, and a human runs an e2e test and explicitly confirms before merge.

## What Makes This Cool

TODOS.md across all of your local projects becomes a single, autonomous work queue. Drop
a well-formed TODO entry into any project's TODOS.md, and within 5 minutes a Claude Code
pipeline picks it up, plans it, implements it, runs a security review, updates docs, and
opens a PR — entirely hands-off. The only manual step left is the one that should stay
manual: confirming the change actually works before it merges. A glance at the kanban
board tells you, per project, "what's it working on right now and what phase is it in" —
without opening a terminal.

## Constraints

- Single user, personal automation — not a multi-tenant or team tool.
- Multiple local project repos, each independently cron-scanned.
- The **hermes agent framework runs on a remote server**; the user has no local
  environment to inspect its kanban plugin's CLI/API right now. The kanban integration
  must be a thin, swappable adapter so the rest of the orchestrator doesn't depend on
  hermes's exact interface.
- The original plan's package structure (`config.py`, `state.py`, `phases.py`,
  `runner.py`, `watcher.py`, `slack.py`), `configs/phases.yaml`, and the TODOS.md field
  schema (What/Why/Pros/Cons/Context/Depends on/Decisions) are locked in — this session
  extends them, doesn't replace them.
- Cron cadence stays 5 minutes.

## Premises

1. **Kanban target = hermes kanban plugin**, accessed via a `hermes` CLI (same pattern as
   the existing `hermes chan message` Slack integration), with an unknown subcommand
   surface — treated as a thin adapter boundary to confirm later.
2. **One TODO in flight per project**, enforced via the existing per-project lock;
   multiple projects run concurrently.
3. **Never auto-merge.** Pipeline halts after Phase 8 (PR open) and persists a
   "ready for review" state. A separate, explicitly-invoked command performs Phase 9
   (merge), gated on the user's e2e confirmation.
4. **Branch version ≠ release version.** The `{version}` in `feat/{version}-{slug}`
   describes *what's being attempted*; `VERSION`/`CHANGELOG.md` are only updated when
   Phase 9 (merge) runs.
5. **TODOs need stable IDs**, independent of title or list position, so dependencies,
   branch names, kanban card links, and rerun/attempt tracking don't drift when
   TODOS.md is hand-edited.
6. **Dependencies are within-project only.** `Depends on` references another `TODO-<n>`
   in the *same* project's TODOS.md. Cross-project dependencies are out of scope — if a
   future need arises, it requires a project-qualified ID scheme and is a separate
   design pass.
7. **The per-project lock is held from selection through merge/reject/abandon**, not
   just through Phase 8. "One TODO in flight per project" means exactly one TODO occupies
   that project's lock at a time, end to end — `--auto` will not start a new TODO for a
   project that has a `ready_for_review` record pending. `pipeline-watch --merge`
   acquires the same lock as `--auto` (mutual exclusion, not a separate lock).
8. **Kanban tracks the in-flight task, not the backlog.** The kanban board is not a
   mirror of TODOS.md — it shows, per project, the single TODO currently checked out
   (if any) and what phase it's in. Completed/pending/on-hold TODOs that aren't the
   active one are not represented on the board.

## Cross-Model Perspective

Ran a Codex cold-read (`gpt-5.5`, high reasoning) against this refined scope. Key points:

- **Steelman:** "a local, deterministic release agent: it turns TODOS.md into the source
  of truth for work selection, branch execution, review handoff, and audit trail...
  aggressive automation before review, conservative at merge."
- **Selection:** proposed a weighted scoring formula (priority + effort + immediate/
  downstream-unblock bonuses + aging), with explicit handling for dependency cycles, an
  all-blocked state, and starvation of low-priority/XL items.
- **Versioning:** branch versions should be *attempt IDs* (`feat/v{base}+todo.{id}.{n}-{slug}`),
  not release versions. Final semver bump (patch/minor/major) is decided from the TODO's
  `Decisions` metadata only at merge time.
- **Kanban:** originally proposed a `KanbanClient` interface mirroring *all* TODOs as
  cards through domain states (`todo_open`/`todo_in_progress`/`todo_failed`/
  `todo_ready_for_review`/`todo_done`), with a local outbox for non-blocking sync.
  This session narrows that scope further (Premise 8): one card per project, tracking
  only the currently-running task — Codex's domain-state vocabulary and outbox/
  non-blocking pattern still apply, just to a single card per project instead of one
  per TODO.
- **Merge:** confirmed stop-after-Phase-8 is right; recommended a separate
  `pipeline-merge <project> <todo_id>` command (never invoked by `--auto`), with a
  persisted state record (branch, head_sha, PR url, phase results, kanban card id,
  `merge_status: pending|merged|rejected|abandoned`).
- **Flagged risks:** full bidirectional kanban sync is over-engineered for a
  single-user tool (a one-way mirror is enough); **TODO identity** (stable IDs) is
  under-specified in the original plan and will cause drift if not addressed now.

Claude's take: agreed on architecture (stop-after-8, one-way non-blocking kanban,
attempt-based branch versions, stable TODO IDs). Disagreed on scope of the selection
algorithm — Codex's full multi-term weighted formula is more machinery than today's
backlog size needs. A simpler ordering (priority → effort → "does this unblock
anything" boolean → declaration order) gets most of the value with much less code to
maintain and tune. On kanban, the user further narrowed scope mid-review: track only
the active task per project, not the full backlog — an even smaller surface than either
model originally proposed.

## Approaches Considered

### Approach A: Pragmatic (recommended)

- **Selection:** build the dependency graph from *all* TODOs in the project's
  TODOS.md (including `[x]` done and `[~]` on-hold — `[x]` deps count as satisfied,
  `[~]` deps count as **unsatisfied**, so anything depending on an on-hold TODO is
  ineligible until it's reactivated). Detect cycles first: any TODO in a cycle is
  marked ineligible with reason `dependency_cycle` and surfaced via the existing Slack
  notification on that tick. From the remaining `[ ]` TODOs, filter to those with all
  `Depends on` satisfied, in a project that isn't locked. Sort eligible TODOs by:
  `(Priority asc, Effort asc, unblocks_something desc, TODO-<n> ascending)` —
  "declaration order" means `TODO-<n>` order *within that project's* TODOS.md (selection
  is per-project; there is no global cross-project ordering). `unblocks_something` is a
  boolean — true if completing this TODO would make at least one other `[ ]` TODO's
  dependencies fully satisfied. If no TODO is eligible, log "no eligible TODO" for that
  project and take no action this tick. No tunable weights.

  *Effort-ascending is a known, accepted tradeoff*: at equal priority, S/M items go
  before L/XL items. For a small personal backlog this is fine — XL items still run once
  they're the only eligible item at their priority tier, or once higher-priority work is
  exhausted. If XL starvation becomes a real problem, that's the trigger to revisit
  toward Approach B's aging/downstream-unblock terms.

- **Versioning:** `feat/{base_version}-{slug}`, where `base_version` is the current
  `VERSION` file contents and `{slug}` is derived from the TODO title (this intentionally
  does **not** adopt Codex's `+todo.{id}.{n}` embedding — TODO IDs are tracked in state/
  kanban, not the branch name, to keep branch names short and human-readable). On rerun
  of the same `TODO-<n>` (a *new attempt* — defined as: the previous attempt's branch was
  abandoned/rejected via Phase 9, not a checkpoint-resume of an in-progress attempt via
  `from_phase`), append `-attemptN` (N starting at 2). `VERSION`/`CHANGELOG.md` are
  untouched by `--auto`; only `pipeline-watch --merge` (Phase 9) writes them, using the
  semver bump rule below.

- **Kanban:** a `KanbanClient` interface with `NullKanbanAdapter` (no-op; the default and
  only active adapter until the hermes interface is confirmed) and a
  `HermesKanbanAdapter` placeholder that raises `NotImplementedError` rather than
  guessing at `hermes kanban` subcommands — avoids writing speculative shell-out code
  against an undefined surface. The interface is scoped to **one card per project**
  (Premise 8):

  ```python
  class KanbanClient:
      def set_active_task(project, todo_id, title, phase) -> SyncResult
      def update_phase(project, phase, status) -> SyncResult   # "Phase 4: Development — running"
      def clear_active_task(project) -> SyncResult              # TODO done / abandoned / rejected
  ```

  The orchestrator calls `set_active_task` when a TODO is selected, `update_phase` as
  the runner advances through `phases.yaml`, and `clear_active_task` when the TODO
  reaches `[x]` (merged) or its `ready_for_review` record is rejected/abandoned. There is
  no card for `[ ]`/`[~]` backlog items — only the one active task per project, if any.
  Sync failures append to `state/kanban_outbox.jsonl` (capped at 500 entries, oldest
  dropped first) and retry on the next tick; kanban sync never blocks the pipeline.

- **Merge:** pipeline halts after Phase 8, persists a `ready_for_review` record (branch,
  PR url, phase summaries, kanban card id, `merge_status: pending`) and **keeps the
  project lock held** (see Premise 7) — the kanban card stays showing the active task at
  its last phase ("Phase 8: Finish Branch — ready for review"). A new `pipeline-watch
  --merge <project> --todo <id>` command performs Phase 9 under that same lock:
  prompts/checks for e2e confirmation, decides the semver bump from the TODO's
  `Decisions` field (mapping TBD — see Open Questions), updates `VERSION`/`CHANGELOG.md`,
  merges, sets `merge_status: merged`, calls `clear_active_task`, and releases the lock.
  If the user rejects/abandons instead, `merge_status` is set to `rejected`/`abandoned`,
  the branch is left in place for inspection, `clear_active_task` is still called (the
  board shouldn't show stale work), and the lock is released so `--auto` can pick the
  next TODO.

- **Stable TODO IDs:** `TODO-<n>` is assigned **once, by the `todos-manager` skill, from
  a monotonically-increasing per-project counter file** (e.g.
  `.hermes/todo_id_counter`) — never reused, never renumbered, regardless of completion
  or removal. This is what makes `TODO-<n>` safe to use in dependencies, branch attempt
  tracking, kanban card linkage, and lock/state files even after hand-edits to TODOS.md.

- **Effort:** M. Reuses nearly all of the original plan's modules; adds `selection.py`,
  `kanban.py`, and the `pipeline-merge` entrypoint.

**Pros:**
- Small, easy to reason about and test; matches the size of a personal backlog.
- The single-card-per-project kanban scope (Premise 8) is even cheaper than a full
  TODO mirror — three calls (`set_active_task`/`update_phase`/`clear_active_task`)
  instead of syncing every status transition for every TODO.

**Cons:**
- The "unblocks_something" boolean is coarser than a true critical-path analysis — a
  TODO that unblocks a *chain* of high-priority work isn't weighted higher than one that
  unblocks a single low-priority item. Acceptable for a small backlog; revisit if it
  isn't.
- The kanban board can't answer "what's queued next" or "what's done" — only "what's
  running now." If that visibility is wanted later, it's an additive change (a second
  `KanbanClient` method), not a redesign.

### Approach B: Full scoring engine

Codex's complete weighted formula (priority + effort + immediate/downstream-unblock +
aging, all configurable via YAML), same versioning/kanban/merge model as A.

**Pros:**
- Tunable; scales gracefully if the backlog grows to dozens of TODOs across many
  projects with deep dependency chains.

**Cons:**
- More code to write and test; weights need tuning over time with little data to tune
  them against right now. Premature for the current backlog size.
- Effort: L.

### Approach C: Kanban-driven selection (lateral)

Push "what's next" onto the hermes kanban plugin itself (`hermes kanban next
<project>`); TODOS.md becomes a generated mirror of the board rather than the primary
input.

**Pros:**
- Avoids duplicating scheduling logic that may already exist in hermes.

**Cons:**
- Makes the core 5-minute loop hard-depend on a remote service whose interface and
  uptime characteristics are currently unknown. Riskier given Premise 1 (interface
  unconfirmed). Effort: S-M for this repo, but carries integration risk.

## Recommended Approach

**Approach A (Pragmatic).** It delivers all five refinements (selection, versioning,
kanban, concurrency, merge-gating) with minimal new surface area, keeps the kanban
integration genuinely optional/non-blocking and now minimal in scope (one card per
project, active task only — Premise 8), and leaves a clean seam to upgrade to Approach
B's scoring formula or a richer kanban view later if needed — without having to touch
the other pieces.

## Open Questions

- **hermes kanban CLI interface** — exact subcommands and JSON shapes are unknown until
  the user has access to the remote hermes environment. `HermesKanbanAdapter` should be
  built against the `KanbanClient` interface first (`set_active_task`/`update_phase`/
  `clear_active_task`), with `NullKanbanAdapter` as the default, and wired up once the
  interface is confirmed.
- **Semver bump mapping** — Phase 9 reads the bump (patch/minor/major) from the TODO's
  `Decisions` metadata, but the exact mapping rules (e.g., does "Security Review:
  required" always imply at least patch?) are not yet defined and should be nailed down
  when `pipeline-merge` is implemented — Phase 9's *responsibility* is settled, only the
  mapping table is open.
- **"Next iteration" dependency** — Approach A's boolean `unblocks_something` may prove
  too coarse once there are deeper dependency chains; if the orchestrator starts picking
  TODOs that unblock only trivial follow-ups while a high-priority chain sits idle,
  revisit toward Approach B's downstream-unblock scoring.

## Success Criteria

- `pipeline-watch --auto` selects the next TODO per project according to the documented
  ordering (deps satisfied, project unlocked, sorted by priority/effort/unblock/order).
- New branches are named `feat/{base_version}-{slug}` (with `-attemptN` on rerun).
- The hermes kanban board shows, per project, the currently-running TODO and its current
  phase (via `set_active_task`/`update_phase`), and is cleared (`clear_active_task`) when
  the TODO is merged, rejected, or abandoned. Backlog/done/on-hold TODOs are not
  represented on the board. Kanban sync failures queue in `state/kanban_outbox.jsonl`
  and retry, never blocking the pipeline.
- After Phase 8, the pipeline halts, persists a `ready_for_review` record, and takes no
  further automatic action on that TODO.
- `pipeline-watch --merge <project> --todo <id>` is the only path that updates
  `VERSION`/`CHANGELOG.md` and performs the merge, and only runs on explicit invocation.

## Distribution Plan

No separate distribution — this is the package itself. Existing plan already covers
local install (`pip install -e ./hermes-pipeline`) and cron registration; no changes
needed for this refinement.
